# solve-workflow 详细参考

本文件提供 `solve-workflow` 的阶段输出模板、判定细则与红旗清单。  
`SKILL.md` 负责协议和门控，**具体怎么输出、怎么判定、怎么提问** 以本文件为准。

---

## 阶段 1：分析问题

### 1.1 明确问题

适用场景：用户还没明确问题，或需要先对齐范围。

**输出结构**

```text
【问题复述】我理解您的问题是：...
【关键要素】目标：... / 约束：... / 背景：... / 期望结果：...
【Scope 拆解】（若适用）模块、依赖、顺序、首个子问题：...
【需要确认的点】
1. ...
2. ...
请确认我的理解是否正确。
```

**要求**

- 只复述意图与边界，不写根因。
- 若用户给了 `@文件`、代码片段、路径或符号，可只读锚点做澄清，不得扩展到其他文件。
- 若问题涉及多个独立子系统，先拆解 Scope，再进入技术分析。

### 1.2 技术分析

**执行顺序**

1. 存在性验证
2. 现象描述
3. 代码定位
4. 根因分析
5. 行业通病评估
6. 影响范围

**存在性验证判定**

| 结论 | 处理方式 |
|------|----------|
| ✅ 问题存在 | 继续后续分析 |
| ❌ 问题已不存在 | 报告已变更或已修复，停止分析 |
| ⚠️ 描述与代码不符 | 报告实际发现，回到 1.1 重新对齐 |

**行业通病评估**

- 若根因触及平台、协议、语言或标准的硬限制，先做行业通病判断。
- 若有疑虑，使用 WebSearch 调研是否属于业界公认难题。
- 若结论为“无可行解”，中断工作流，不进入阶段 2。

**输出结构**

```text
【存在性验证】...
【问题现象】...
【相关代码定位】...
【根因分析】...
【行业通病评估】...
【影响范围】...
```

## 阶段 2：探索方案

### 目标

- 给出 2～5 个可行方案。
- 避免过度设计，遵守 YAGNI。
- 明确推荐项。

### 输出顺序

1. 先给对比表。
2. 再展开每个方案。
3. 最后再给一次对比表，便于回看。

### 对比表字段

| 方案 | 描述 | 优点 | 缺点 | 复杂度 | 推荐度 |
|------|------|------|------|--------|--------|

### 方案展开字段

- 核心思路
- 需要修改的文件 / 模块
- 风险与缓解
- 适用场景
- 一句推荐理由

### 推荐度

- ⭐⭐⭐⭐⭐：首选
- ⭐⭐⭐⭐：强备选
- ⭐⭐⭐：可行
- ⭐⭐：勉强
- ⭐：不推荐

## 阶段 3：审查方案

### 每轮必须覆盖

1. 核心逻辑完整性
2. 风险可控性
3. 实现可行性
4. 代码 / 项目规范符合度

### 审查结论

| 结论 | 含义 |
|------|------|
| ✅ 通过 | 可进入阶段 4 |
| ❌ 不通过 | 需要优化后重审 |

### 不通过判定

满足任一项即可判为不通过：

- 无法完整覆盖阶段 1 的根因
- 风险过高且没有缓解措施
- 改动范围或依赖关系不清晰
- 与项目现有模式冲突
- 可能引入新 bug 且未处理副作用

### 审查报告字段

```text
【选定方案】...
【方案解析】核心逻辑：... / 涉及文件与模块：... / 关键实现点：...
【风险识别】...
【方案处理建议】建议 修改/完善/优化，理由：...
```

---

## 阶段 4：制定计划

### 要包含的内容

1. 目标方案回顾
2. 文件修改清单
3. 修改顺序
4. 预期影响与风险点

### 输出模板

```text
【目标方案回顾】...
【文件修改清单】
1. 文件：...，位置：...，改动：...
2. 文件：...，位置：...，改动：...
【修改顺序】...
【预期影响】范围：... / 风险点：...
```

---

## 阶段 5：执行计划

### 要求

- 严格按计划执行
- 每次修改后说明完成内容
- 有偏离计划时说明原因
- 结束后给出执行报告

### 执行报告建议字段

- 已修改文件列表
- 关键改动说明
- 建议测试步骤
- 偏离计划说明
- 验证要点 / 自查清单

---

## 阶段 6：检查验证

### 检查维度

1. 阶段 1 的目标是否达成
2. 与阶段 4 的计划是否一致
3. 是否执行了需要执行的测试
4. 是否存在流程漏洞或遗漏

### 输出结构

```text
【检查结果】
- 阶段1目标达成情况：...
- 与阶段4计划对比：...
- 验证要点 / 测试结论：...
- 逻辑与整体流程审查：...
```

### 测试规则

- 如果计划或执行报告里写了测试，就应实际跑。
- 如果环境限制导致不能跑，要明确告诉用户需要手动执行什么。

---

## 阶段 7：回顾总结

### 输出结构

```text
【改进建议】
- 可固化做法：...
- 下一步建议：...
- 遗留风险：...
```

### 额外动作

- 可以询问是否需要生成总结文档。
- 只有用户明确要求时，才进入下一轮修改。

---

## 通用提问格式

当信息不足时，优先用二选一或简答题推动决策：

```text
1. 题干
   - A 选项一
   - B 选项二
2. 下一题
   - A 选项一
   - B 选项二
```

用户可以直接回复 `1A, 2B`。

---

## 红旗清单

- 无锚点时泛扫仓库
- 1.1 阶段写根因或修复建议
- 跳过存在性验证
- 未选方案就进计划
- 审查不通过仍强行进入执行
- 验证阶段不跑已声明的测试
