# 七阶段问题解决工作流

## 与戴明环（PDCA）的对应

| PDCA | 阶段 | 说明 |
|------|------|------|
| **Plan** | 阶段 1～4 | 明确问题、分析问题、评估方案、制定计划 |
| **Do** | 阶段 5 | 执行计划 |
| **Check** | 阶段 6 | 检查验证 |
| **Act** | 阶段 7 | 回顾改进 |

执行完成后进入阶段 6；若阶段 7 结论为「未达标需再改」，可回到「分析问题」或「评估方案」开启下一轮循环。

---

## 通用原则

### 主动提问

各阶段进行中，若当前信息不足以保证输出质量，应先提出 1～3 个关键问题，待用户补充后再继续。

**提问格式**：
```
1. 题干（一句话说清问题）
   - A 选项一
   - B 选项二
2. 下一题题干
   - A 选项一
   - B 选项二
```

**简答约定**：用户可直接回复「1A, 2B」等形式，AI 按题号与选项字母解析后继续。

### 临时内容管理

在阶段 2 或阶段 5 中添加临时日志、打点等诊断代码时：

**添加时（两步）**：
1. 在对话中声明：`📝 添加临时内容：文件路径:行号 - [用途]`
2. 在代码中标记：`// TEMP-DEBUG: [用途]，阶段6验证后删除`

**清理时（阶段 6）**：用 Grep 搜索 `TEMP-DEBUG`，逐一清理后输出确认。

---

## 阶段1：明确问题

> 原则：通过对话确保对问题的理解一致，不进行代码操作

1. **问题复述** - 用自己的话重新描述用户的问题
2. **关键要素提取** - 目标、约束、背景、期望结果
3. **疑问点列出** - 列出需要进一步确认的地方
4. **等待用户确认**

### 输出格式

```
【问题复述】我理解您的问题是：...
【关键要素】目标：... / 约束：... / 背景：... / 期望结果：...
【需要确认的点】
1. [题干] A [...] B [...]
2. [题干] A [...] B [...]
请确认我的理解是否正确。
```

**工具限制**：禁止使用任何代码读取或修改工具

---

## 阶段2：分析问题

> 原则：只读分析为主，不修改实现逻辑；为辅助定位而加的临时日志允许，须按「临时内容管理」流程处理

1. **信息补全（如需要）** - 缺少关键信息时先提问
2. **问题现象描述** - 复现条件和步骤
3. **相关代码定位** - 文件路径+行号、关键函数/类
4. **问题根因分析** - 数据流和调用链分析
5. **影响范围评估** - 受影响的模块/功能

**工具限制**：
- ✅ Read/Grep/SemanticSearch 收集信息
- ❌ Edit/Write 修改代码（临时日志除外，须标记 TEMP-DEBUG）
- ❌ Bash 执行改变实现逻辑的命令

---

## 阶段3：评估方案

> 原则：基于阶段2的分析，提供2-5个解决方案

### 输出格式

| 方案 | 描述 | 优点 | 缺点 | 复杂度 | 推荐度 |
|------|------|------|------|--------|--------|
| 方案1 | ... | ... | ... | 低/中/高 | ⭐⭐⭐⭐⭐ |
| 方案2 | ... | ... | ... | 低/中/高 | ⭐⭐⭐ |

### 每个方案包含

- 核心思路（1-2句话）
- 需要修改的文件/模块
- 实施难度评估、潜在风险、适用场景

**工具限制**：禁止使用 Edit/Write；可使用 Read 查看代码细节

---

## 阶段4：制定计划

> 原则：详细的、可执行的修改计划，只输出计划文本，不执行代码修改

1. **目标方案回顾** - 选定方案的核心思路
2. **文件修改清单** - 文件路径、修改位置、具体改动描述
3. **修改顺序** - 考虑依赖关系的执行顺序

### 输出格式

```
【目标方案回顾】采用方案X：...
【文件修改清单】
1. 文件：xxx/yyy/zzz.js，位置：function abc()，改动：...
2. 文件：xxx/yyy/aaa.js，位置：class DEF.methodXYZ()，改动：...
【修改顺序】1. xxx/yyy/zzz.js（依赖项）→ 2. xxx/yyy/aaa.js（调用方）
【预期影响】范围：... / 风险点：...
```

**工具限制**：禁止使用 Edit/Write/Bash 修改代码；可使用 Read 确认细节

### 更新计划（子阶段）

当用户说「更新计划」「修订计划」「修改计划」时，按本阶段执行，额外标注**变更对比**和**变更原因**。

---

## 阶段5：执行计划

> 原则：严格按计划执行，完成后确认

### 执行流程

1. 按计划顺序修改文件
2. 每次修改后说明完成内容
3. 全部完成后输出执行报告

### 执行报告

- 已修改文件列表、关键改动说明
- 建议的测试步骤
- 是否有偏离计划的调整（如有，说明原因）
- 验证要点/自查清单（便于阶段6使用）

**工具权限**：✅ 允许使用 Edit/Write/Bash；使用 TodoWrite 跟踪进度

---

## 阶段6：检查验证（Check）

> 原则：只输出检查结果，不输出改进建议；改进建议在阶段 7

1. **目标达成情况** - 是否达成阶段 1 的期望结果
2. **与计划对比** - 与阶段 4 的计划对比
3. **验证与测试** - 可引用阶段 5 执行报告
4. **逻辑与流程审查** - 检查是否存在漏洞或遗漏

### 输出格式

```
【检查结果】
- 阶段1 期望结果达成情况：...
- 与阶段4 计划对比：已做到 ... / 未做到 ...，原因 ...
- 验证要点/测试结论：...
- 逻辑与整体流程审查：...

【清理诊断】是否需要清理本次会话中添加的临时内容？
- A 是，请清理
- B 否，暂时保留
```

**工具限制**：禁止使用 Edit/Write（清理诊断子阶段除外）

### 清理诊断（子阶段）

触发方式：用户选择「A 是，请清理」或直接说「清理诊断」「移除调试」

**流程**：
1. 用 Grep 搜索 `TEMP-DEBUG`
2. 无结果 → 提示「无需清理」
3. 有结果 → 逐一清理，再次搜索确认无遗漏
4. 输出：`已清理临时内容：1. xxx:行号 - 用途 / 共清理 X 项`

---

## 阶段7：回顾改进（Act）

> 原则：只做总结与建议，不强制改代码；若用户明确要求根据改进点再改，则进入新一轮「制定计划 → 执行计划」

1. **固化有效做法** - 文档、注释、习惯等如何保留
2. **未达标时的下一步** - 是否进入下一轮
3. **收尾** - 记录遗留风险与后续改进点

### 输出格式

```
【改进建议】
- 可固化做法：...
- 建议：进入下一轮 / 收尾。若收尾，遗留与后续改进：...
```

**工具限制**：禁止使用 Edit/Write，除非用户明确要求进入下一轮修改

---

## 输出详细度控制（自适应）

- 简单问题（单文件修改）：精简输出
- 中等复杂度：标准输出
- 高复杂度（多模块联动）：详细输出 + 代码示例
