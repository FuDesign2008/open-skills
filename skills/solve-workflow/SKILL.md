# 七阶段工作流（精简版）

**约定**：阶段 1～4 **只读**；**`Edit/Write`** 主要在 **阶段 5**。**阶段 5 完成 → 阶段 6**。循环：阶段 7 未达标 → 回到 1 / 2 / 3。

## 详细参考

> 先看本文件确认阶段与门控；执行细节见 `reference.md`；对照样例见 `examples.md`。

- `reference.md`：阶段 1～7 的详细输出模板、判定细则、红旗清单
- `examples.md`：每阶段的好/坏输出示例、自动/手动模式样例

## 最小硬约束

1. **阶段 1.1 先对齐问题，再谈技术**：只做复述、要素、Scope、确认问题；若用户给了锚点，可只读锚点做澄清，但不得扩展到无关仓库。
2. **阶段 1.2 先做存在性验证**：先确认问题是否仍存在，再输出现象、定位、根因、通病评估、影响面。
3. **阶段 2 必须给对比表与推荐**：至少 2 个方案，明确推荐项；微任务可缩写，但不能省掉推荐。
4. **阶段 3 不通过不能进 4**：自动模式最多 3 轮，超限必须暂停。
5. **阶段 4 必须列文件、顺序、风险**：不是只说“我会改”，而是要能按计划执行。
6. **阶段 6 若计划里写了测试，就要验证或明确告诉用户怎么测**。
7. **阶段 7 只做总结与下一步建议，不改代码**。

## 门控（改仓库前自问）

| 情形 | 改仓库 |
|------|--------|
| 👤 手动 且非阶段5 且无 **显式合并授权**、无 **快速通道** | ❌ |
| 阶段5 或 已授权 / 已声明快速通道 | ✅（遵守项目规则） |
| 🤖 自动 | 按阶段表，通常阶段5可写 |

**显式合并授权**（同条消息含「只分析/不要改代码/先别改」则 **不适用**「直接修」类）：「制定计划并执行计划」→4 后可直接 5；「分析完直接修」「确认后帮我改好」→1.2 后可跳过 2～4 进 5（首次改前注明「已获权跳过 2～4」）；「自动模式」→走 🤖。

**快速通道**：用户须 **明说**「快速通道」或「跳过方案与审查」+ 1.2 已澄清 + 真·微任务 + 非高危；**首次 Edit 前**输出【快速通道声明】（跳过哪些阶段、理由、风险）。

**严格流程**：说「严格流程」「禁止跳步」「按阶段来」等 → 禁止隐式跳步；**1.1 选项≠执行授权**，下一动默认 **1.2**；**1.2～4** 单次交付后 **默认停** 等用户下一步；每段结束 **声明阶段号**。摘要/换题后 **不假设** 阶段；「新问题从阶段1开始」→重置授权。

**🔌 增强能力**：
- **扫能力**：启动时用系统提示里的 skill 列表等元数据扫一次即可；**禁止**为扫能力而无目的 Read 仓库。
- **读锚点**：**1.1** 仅可对用户消息里已有的锚点（`@` 文件、粘贴片段、已写路径/符号/关键词）做窄范围 Read/Grep 用于澄清；存在性验证与根因链仍以 **1.2** 系统做完。
- **skill 可选用**：匹配到 debug / brainstorm / review / plan / execute / tdd / build-fix / verify 类 skill 时在对应阶段可选用，**不改变**阶段门控与只读约束；失败则忽略。

## 模式

- **👤 手动**（默认）：1.1→确认→1.2→**停**→2→…→4→确认→5→6→7（各步间按「严格」与否停顿）。
- **🤖 自动**（消息含「自动」）：跳过 1.1；1.2→2→3（审查≤3 轮，超限停）→4→5→6→7 连贯推进。

---

## 阶段 1：分析问题

**禁止 Edit/Write**（例外：1.1 仅可读的边界见下）。

### 1.1 明确问题（👤 须确认；🤖 跳过）

复述、要素、疑问点、Scope。

**只读（为澄清，非定因）**：仅当 **用户消息里已有锚点**（`@` 文件、粘贴片段、已写路径/符号/关键词）时，可对该锚点做 **窄范围** Read/Grep，用于对齐需求、范围、复现入口。**禁止**无锚点的仓库漫游、大目录枚举、「顺手扫一遍」。**1.1 输出** 仍偏「复述+要素+待确认+请确认」；**不写技术根因、不下技术定论**（留给 1.2）。

### 1.2 技术分析（存在性验证 → 现象 → 定位 → 根因 → 行业评估 → 影响面）

- **存在性**：先做。**非代码任务** 标注【非代码任务】、自洽性替代「代码是否存在」。**有代码** 再 Read/Grep。不存在/不符 → 停，等用户或回 1.1。
- **行业通病 (3.5)**：疑为平台/协议/语言硬限制时，**有 WebSearch** 可搜；**无** → 【行业评估·无联网】、标不确定性，**勿编造**、无证据 **慎** 判 🚫 无解。🚫→出简要评估报告并 **中断**；⚠/✅ 继续。
- **工具**：Read/Grep/Search/WebSearch（3.5）；❌ Edit / 改逻辑 Bash。

**输出**：1.1 用简短「复述+要素+待确认+请确认」（若已窄读锚点，可一句交代「已见入口/片段」**不写根因**）；1.2 先给存在性结论，再给根因与通病结论。

**禁止**：**无用户锚点** 的 Read/Grep；1.1 输出里写根因或定论；**仅**回复「1A,2B」后 **同轮/紧随** `Edit`（除非授权/快速通道）。

---

## 阶段 2：探索方案

**禁止 Edit**。2～5 个方案 + **推荐**一项；👤 等「方案N/跟推荐」再进 3。🤖 自选自审。YAGNI。可选用 brainstorming。

**输出顺序**：① **对比表** → ② 各方案展开 → ③ **同表再输出**（与①列一致，便于上扫/下回顾）。

**表**（①③同构）：| 方案 | 概要 | 优点 | 缺点 | 复杂度 | 推荐度 |

- **复杂度**：低 / 中 / 高。
- **推荐度**（主判断依据）：⭐⭐⭐⭐⭐ 首选 ≈ ⭐⭐⭐⭐ 强备选 ≈ ⭐⭐⭐ 可行 ≈ ⭐⭐ 勉强 ≈ ⭐ 不推荐；展开里用**一句**点明星级理由。
- **微任务**可 2 方案；极短时可把「优/缺」合并为一列「权衡」，**须保留推荐度列**且①③结构一致。

---

## 阶段 3：审查方案

**禁止 Edit**。每轮覆盖：逻辑完整性、风险、可行性、规范。✅ / ❌；❌→优化再审。**🤖 至多 3 轮** 审查循环后须停。👤 等「通过/修改方案/重选」。可选用 code-review。

---

## 阶段 4：制定计划

**禁止 Edit / 改仓库 Bash**。回顾方案、文件清单、顺序、风险。**👤** 确认后进 5；**🤖** 自动进 5。「修订计划」→本阶段并写变更说明。

---

## 阶段 5：执行计划

✅ Edit / Bash。**子 Agent 并行**：主会话仍算阶段 5，**合并与 6～7 仍由主会话收尾**。可选用 TDD/execute/build-fix。执行报告：改了啥、测试建议、偏离说明。**顺**→进 6；卡壳→问用户。

---

## 阶段 6：检查验证

**禁止 Edit**；可跑测试 Bash。对照阶段1期望与阶段4计划；有测试则跑或提示用户跑。输出【检查结果】要点。可选用 verification。

---

## 阶段 7：回顾总结

**禁止 Edit**（除非用户明确下一轮）。改进建议、是否再迭代、遗留风险；可问是否写总结文档。

---

## Red Flags（合并速查）

| 踩雷 | 纠正 |
|------|------|
| 无锚点泛扫 / 为扫能力读仓库 / 1.1 写根因 | 元数据扫能力；1.1 只读须带锚点且窄范围；根因在 1.2 |
| 跳过存在性或通病门控 | 按 1.2 顺序执行 |
| 1～4 写仓库 / 无授权的隐式跳步 | 仅 5 或授权/快速通道 |
| 把 1.1 选项当「执行计划」 | 默认进 1.2 |
| 👤 未选方案就审查、未过审就进 4 | 按门推进 |
| 🤖 审查>3 轮不停 | 必须停 |
| 无联网却下死「行业定论」🚫 | 【无联网】+ 慎 🚫 |
| 增强 skill 失败或突破只读 | 忽略失败；只读不变 |

**详略**：简单任务可 **缩写篇幅**（如 2 个方案），**不得**借此跳过 👤 门控；省阶段须 **授权** 或 **快速通道**。

**自检**：每次 `Edit` 前：**是阶段 5 或有权跳步？**「自适应」只缩字数，**不**缩阶段。
