# 七阶段问题解决工作流

七阶段 PDCA 流程，阶段 1～4 只读不写，执行计划前须用户确认。

## 与戴明环（PDCA）的对应

| PDCA | 阶段 | 说明 |
|------|------|------|
| **Plan** | 阶段 1～4 | 分析问题（含明确问题）、探索方案、审查方案、制定计划 |
| **Do** | 阶段 5 | 执行计划 |
| **Check** | 阶段 6 | 检查验证 |
| **Act** | 阶段 7 | 回顾总结 |

执行完成后进入阶段 6；若阶段 7 结论为「未达标需再改」，可回到「分析问题」、「探索方案」或「审查方案」开启下一轮循环。

### 阶段速查

| 阶段 | 名称 | 关键约束 |
|------|------|----------|
| 1 | 分析问题 | 1.1 可读引用代码→确认→1.2 可读 |
| 2 | 探索方案 | 禁止 Edit/Write |
| 3 | 审查方案 | 禁止 Edit/Write |
| 4 | 制定计划 | 禁止 Edit/Write/Bash |
| 5 | 执行计划 | 允许 Edit/Write/Bash |
| 6 | 检查验证 | 禁止 Edit；允许 Bash 执行测试 |
| 7 | 回顾总结 | 禁止 Edit（除非进入下一轮） |

---

## 调用约定

- **触发词**（以 description 为准）：明确问题、分析问题、探索方案、审查方案、制定计划、执行计划、检查验证、回顾总结；继续分析、深入分析、修改方案、完善方案、优化方案、更新计划、修订计划、修改计划
- **命令形式**：`/solve-workflow xxx`、`/solve xxx` 等，`xxx` 为后续内容
- **默认行为**：当 `xxx` 不包含上述任一触发词时，默认进入阶段 1（分析问题），将 `xxx` 视为待分析的问题描述
- **匹配规则**：`xxx` 中包含触发词即视为命中，无需精确匹配

---

## 通用原则

### 调查优先、证据驱动

1. **先调查再发言** - 对于不确定的问题，先调查再给出建议或方案；没有调查就没有发言权。
2. **证据说话** - 尽量以数据、事实和证据说话，避免虚空假设。

### 主动提问

各阶段进行中，若当前信息不足以保证输出质量，应先提出 1～3 个关键问题，待用户补充后再继续。

**提问格式**：
```
1. 题干（一句话说清问题）
   - A 选项一
   - B 选项二
2. 下一题题干
   - A 选项一
   - B 选项二
```

**简答约定**：用户可直接回复「1A, 2B」等形式，AI 按题号与选项字母解析后继续。

**一次一问**：若问题复杂或易产生歧义，可改为一次一问，降低认知负担。

## 阶段1：分析问题

> ⚠️ 本阶段禁止 Edit/Write。仅输出分析结论，不修改任何文件。

> 原则：先明确问题再技术分析；只读分析为主，不修改实现逻辑

### 1.1 明确问题（小节）

**必须先完成本小节并获用户确认后，才能进入 1.2 技术分析。**

1. **问题复述** - 用自己的话重新描述用户的问题
2. **关键要素提取** - 目标、约束、背景、期望结果
3. **疑问点列出** - 列出需要进一步确认的地方
3.5 **Scope 拆解**（若适用）- 若问题涉及多个独立子系统（如「聊天 + 文件存储 + 计费」），先协助拆解：独立模块、依赖关系、建议处理顺序，再对首个子问题进行分析。**本步骤仍在本小节内，禁止主动探索代码；用户已引用的内容可读取**，拆解主要依据用户描述。
4. **等待用户确认**

**工具限制**：禁止 Read/Grep/SemanticSearch，以下情况**例外**：
- 用户消息中含 `@文件路径`（含可选行号，如 `@SKILL.md:83-89`）
- 用户消息中粘贴了代码片段（`` ``` `` 包裹或缩进代码块）
- 用户明确指出了「函数/类名 + 所在文件」的组合

例外时：**仅读取用户直接引用的文件与行号**，不得扩展到其他文件。
读取结果仅用于辅助理解问题，**1.1 输出中不得出现技术分析结论**。

### 1.2 技术分析

1. **问题现象描述** - 复现条件和步骤
2. **相关代码定位** - 文件路径+行号、关键函数/类
3. **问题根因分析** - 数据流和调用链分析
4. **影响范围评估** - 受影响的模块/功能

**工具限制**：✅ Read/Grep/SemanticSearch；❌ Edit/Write；❌ Bash 执行改变实现逻辑的命令

### 输出格式

**1.1 明确问题**：
```
【问题复述】我理解您的问题是：...
（只描述用户的意图与现象，不得出现根因判断或修复建议——技术结论留给 1.2）
【关键要素】目标：... / 约束：... / 背景：... / 期望结果：...
【Scope 拆解】（若适用）模块、依赖、顺序、首个子问题：...
【需要确认的点】
1. [题干] A [...] B [...]
2. [题干] A [...] B [...]
请确认我的理解是否正确。
```

**1.2 技术分析**：现象、定位、根因、影响范围（可结合代码引用说明）

### Red Flags - 禁止行为

- 用户未引用代码，却在 1.1 未完成时主动使用 Read/Grep 探索代码
- 以「用户已经说得很清楚」为由跳过 1.1 小节
- 以「边分析边澄清」为由在 1.1 未完成时主动探索代码
- 以「先看看代码再确认」为由在 1.1 未完成时主动探索代码
- 在 1.1 的「问题复述」或「疑问点」中混入技术分析结论（如根因判断、修复建议）

**违反即违反原则。必须先完成 1.1 明确问题并获确认，再进入 1.2。**

---

## 阶段2：探索方案

> 原则：基于阶段1的分析，提供2-5个解决方案；方案中剔除非必要功能与过度设计（YAGNI）

### 输出格式

1. **开头**：方案对比表
2. **中间**：每个方案详细展开（见下方「每个方案包含」）
3. **结尾**：再次输出方案对比表，便于快速回顾

| 方案 | 描述 | 优点 | 缺点 | 复杂度 | 推荐度 |
|------|------|------|------|--------|--------|
| 方案1 | ... | ... | ... | 低/中/高 | ⭐⭐⭐⭐⭐ |
| 方案2 | ... | ... | ... | 低/中/高 | ⭐⭐⭐ |

### 每个方案包含

- 核心思路（1-2句话）
- 需要修改的文件/模块
- 实施难度评估、潜在风险、适用场景

**工具限制**：禁止使用 Edit/Write；可使用 Read 查看代码细节

---

## 阶段3：审查方案

> 原则：对初步选择的方案进行深度审查，明确其具体内容和风险，再决定是否进入制定计划

1. **方案确认** - 用户确认或选定待审查的方案（如推荐方案）
2. **方案深度解析** - 该方案具体是什么、核心逻辑、涉及范围
3. **风险识别** - 可能存在的技术风险、兼容性风险、维护成本等
4. **方案处理** - 根据审查结果，选择以下之一进行迭代（可多轮）：
   - **修改方案** - 调整思路、修正偏差、规避已识别的风险
   - **完善方案** - 补充遗漏细节、明确边界条件、填补实现缺口
   - **优化方案** - 提升效率、降低复杂度、改善可维护性

### 输出格式

```
【选定方案】方案X：...
【方案解析】核心逻辑：... / 涉及文件与模块：... / 关键实现点：...
【风险识别】
- 风险1：... 影响：... 缓解建议：...
- 风险2：...
【方案处理建议】建议 修改/完善/优化，理由：...
请确认是否满意，或说「修改方案」「完善方案」「优化方案」进行迭代。
```

**工具限制**：禁止使用 Edit/Write；可使用 Read 查看代码细节

### 修改/完善/优化方案（子阶段）

当用户说「修改方案」「完善方案」「优化方案」时，按本阶段执行，在上一轮审查基础上进行对应处理，并重新输出审查结果。

---

## 阶段4：制定计划

> 原则：详细的、可执行的修改计划，只输出计划文本，不执行代码修改

1. **目标方案回顾** - 经审查确认的方案核心思路
2. **文件修改清单** - 文件路径、修改位置、具体改动描述
3. **修改顺序** - 考虑依赖关系的执行顺序

### 输出格式

```
【目标方案回顾】采用方案X：...
【文件修改清单】
1. 文件：xxx/yyy/zzz.js，位置：function abc()，改动：...
2. 文件：xxx/yyy/aaa.js，位置：class DEF.methodXYZ()，改动：...
【修改顺序】1. xxx/yyy/zzz.js（依赖项）→ 2. xxx/yyy/aaa.js（调用方）
【预期影响】范围：... / 风险点：...
```

**工具限制**：禁止使用 Edit/Write/Bash 修改代码；可使用 Read 确认细节

### 更新计划（子阶段）

当用户说「更新计划」「修订计划」「修改计划」时，按本阶段执行，额外标注**变更对比**和**变更原因**。

---

## 阶段5：执行计划

> 原则：严格按计划执行，完成后确认

### 执行流程

1. 按计划顺序修改文件
2. 每次修改后说明完成内容
3. 全部完成后输出执行报告
4. **自动进入阶段 6**：执行顺利、无坑无阻塞时，报告输出后立即进入阶段 6；遇问题或需决策时，先与用户确认

### 执行报告

- 已修改文件列表、关键改动说明
- 建议的测试步骤
- 是否有偏离计划的调整（如有，说明原因）
- 验证要点/自查清单（便于阶段6使用）

**工具权限**：✅ 允许使用 Edit/Write/Bash；使用 TodoWrite 跟踪进度

---

## 阶段6：检查验证（Check）

> 原则：只输出检查结果，不输出改进建议；改进建议在阶段 7（回顾总结）

1. **目标达成情况** - 是否达成阶段 1 明确问题小节的期望结果
2. **与计划对比** - 与阶段 4 的计划对比
3. **验证与测试** - 可引用阶段 5 执行报告；**若有测试相关内容，需执行测试**
4. **逻辑与流程审查** - 检查是否存在漏洞或遗漏

### 测试执行

若阶段 4 计划或阶段 5 执行报告涉及测试（如单元测试、集成测试、手动验证步骤）：

- **AI 可执行**：使用 Bash 运行测试命令（如 `npm test`、`pytest`、`go test`），将结果纳入检查结论
- **AI 无法执行**（无 Bash、环境限制、测试需人工操作）：**明确提醒用户**：「本次修改涉及测试，请自行执行 [具体测试命令/步骤] 验证，确认通过后再收尾」

### 输出格式

```
【检查结果】
- 阶段1 明确问题小节的期望结果达成情况：...
- 与阶段4 计划对比：已做到 ... / 未做到 ...，原因 ...
- 验证要点/测试结论：...（若已执行测试，附上结果；若未执行，附上需人工测试的提醒）
- 逻辑与整体流程审查：...
```

**工具限制**：禁止 Edit/Write；✅ 允许 Bash 执行测试命令

---

## 阶段7：回顾总结（Act）

> 原则：只做总结与建议，不强制改代码；若用户明确要求根据改进点再改，则进入新一轮「制定计划 → 执行计划」；亦可回到「审查方案」重新选定或调整方案。

1. **固化有效做法** - 文档、注释、习惯等如何保留
2. **未达标时的下一步** - 是否进入下一轮
3. **收尾** - 记录遗留风险与后续改进点
4. **可选总结文档** - 输出【改进建议】后，主动询问「是否需要生成总结文档」；若用户需要，则生成总结文档（路径由用户指定或 AI 建议），内容包含：问题复述、方案选择、执行结果、遗留与改进点

### 输出格式

```
【改进建议】
- 可固化做法：...
- 建议：进入下一轮 / 收尾。若收尾，遗留与后续改进：...
```

输出后主动询问：「是否需要生成总结文档？」

**工具限制**：禁止使用 Edit/Write，除非用户明确要求进入下一轮修改

## 常见错误

| 错误 | 后果 | 修正 |
|------|------|------|
| 跳过 1.1 直接读代码 | 误解问题、无效分析 | 先完成明确问题并获确认 |
| 评估/审查/制定计划时修改代码 | 破坏只读约束 | 仅用 Read，禁止 Edit/Write |
| 制定计划后未经确认即执行 | 方向偏差 | 等待用户确认计划后再执行 |
| 「问题不复杂」为由拒绝一次一问 | 歧义未澄清 | 复杂/歧义时改为一次一问 |
| 「用户提了即必要」为由不剔除 | 方案臃肿 | 剔除非必要功能（YAGNI） |

---

## 输出详细度控制（自适应）

- 简单问题（单文件修改）：精简输出
- 中等复杂度：标准输出
- 高复杂度（多模块联动）：详细输出 + 代码示例
