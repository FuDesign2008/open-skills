# Jira Bug 分析（阶段2）

> 本 skill 由 `jira-fix-workflow` 在**阶段2**调用，请勿独立使用。
> 完成后返回 orchestrator 执行**阶段2.5：难度分级**。

**🔌 增强能力集成**：若环境探索发现了「🔍 调试分析」类能力（如 `systematic-debugging`、`debug-workflow`、OMC `debugger` agent 等），在根因分析环节加载并遵循其方法论（假设驱动调查、证据链构建）。未发现则按下述流程执行。

## 原则

- 深度分析，追问「为什么」至少 3 次
- 结构化输出，中文描述
- **[👤 手动]** 信息不足时主动提问，每次只问 1 个最关键的问题
- **工具限制**：✅ Read/Grep/SemanticSearch/WebSearch；❌ Edit/Write/Bash

---

## 分析维度

### 第一步：存在性验证（门控，必须最先执行）

用 Read/Grep/SemanticSearch 搜索与 Jira 描述相关的代码，判断：

| 结论 | 处理方式 |
|------|---------|
| ✅ 问题存在 | 继续后续分析维度 |
| ❌ 问题已不存在 | 报告「当前代码库中未发现该问题，可能已被修复或逻辑已变更」，附相关代码位置，**停止分析，更新 Jira 评论并等待用户确认** |
| ⚠️ Jira 描述与代码不符 | 报告「代码行为与 Jira 描述存在出入」，列出实际发现，**等待用户重新确认问题描述** |

### 代码定位

Grep/Glob/SemanticSearch 组合搜索关键代码（路径:行号+说明）。

### 代码定位后立即执行：已知问题快搜（可选门控）

满足以下任一条件时，在根因分析之前先执行 WebSearch：

- 问题涉及**平台/原生组件**（Android WebView、iOS 控件、浏览器 API 等），且表现为**静默失败**（无报错、调用正确但无响应）
- 问题涉及**宿主环境嵌套**（WebView in App、React Native、Electron 等），且前端代码逻辑看起来完全正确
- 代码层面**找不到任何可疑点**（调用链完整、参数正确），但运行时就是不生效

执行：以「症状关键词 + 平台/框架 + 年份」搜索 StackOverflow、GitHub Issues、官方文档中的已知案例。

| 搜索结论 | 处理方式 |
|---------|---------|
| ✅ 找到已知案例，根因明确 | 输出【已知问题快搜结论】，**直接跳到影响范围**，跳过根因分析步骤 |
| ⚠️ 找到相关讨论，根因部分明确 | 将结论纳入根因分析参考，继续根因分析 |
| ❌ 未找到相关案例 | 静默跳过，继续根因分析 |

### 根因分析

数据流（操作→组件→函数→状态→渲染）+ 调用链，区分直接原因与根本原因。

### 根因分析后立即执行：行业通病评估（门控）

判断根因是否触及平台/语言/协议/标准的固有限制；若有疑虑，用 WebSearch 调研业界现状：

| 结论 | 处理方式 |
|------|---------|
| 🚫 行业公认难题，无可行解 | 输出【行业通病评估报告】，**停止流程，不进入阶段3；在 Jira 写评论说明结论** |
| ⚠️ 存在局限但有绕过方案 | 在阶段3方案探索中将绕过方案列为候选，继续后续分析 |
| ✅ 非行业通病，属可修复问题 | 继续后续分析 |

**行业通病评估报告格式**：
```
【行业通病评估】
- 问题本质：...（根因一句话总结）
- 行业现状：...（已知公开记录、主流框架态度、大厂处理方式）
- 调研结论：该问题属于 [平台限制/协议约束/语言特性/标准规范]，业界目前无可行解
- 建议：接受现状 / 评估替代方案（非修复）/ 与产品对齐预期
流程已中断，不进入方案探索阶段。
```

### 影响范围

功能、平台（Web/Mobile）、模块、连带影响。

### 难度预判

预估改动文件数 + 根因清晰度（清晰/基本清晰/模糊/未知，供阶段2.5使用）。

## 🔬 打点调试（静态分析受阻时，主动升级为运行时调试）

**触发条件（满足任一即触发，在进入阶段2.5前优先处理）：**

- 根因置信度为「模糊」或「未知」——能定位到大概模块，但无法确定具体逻辑或触发路径
- 当前是 `--retry` 重试场景——已基于静态分析修复过一次，但问题仍然存在

> **`--retry` 场景首选策略**：「继续修复」模式意味着静态分析已经做过一次，再次静态分析大概率触达同样的边界。应优先考虑打点调试，用运行时事实重新锚定根因，而不是重复上一轮的分析路径。

**执行步骤：**

1. **打点位置设计**：基于当前代码定位，识别 2-5 个关键节点
   - 函数入口/出口：确认是否被调用、调用频率、调用顺序
   - 状态变更点：记录变更前后的值
   - 数据流转点：追踪数据在模块间传递时的实际取值

2. **生成打点代码**（示例格式，按项目语言调整）：
   ```js
   console.log('[DEBUG-<位置标识>]', { key: value, timestamp: Date.now() })
   ```
   要求：含位置标识（便于日志搜索定位）、含关键变量、含时间戳

3. **操作指引**：明确告知用户在哪些文件/行添加打点、如何触发问题场景、在哪里查看日志

4. **等待日志**：⛔ 停止，等用户执行并提供日志输出

5. **日志分析**：收到日志后，分析实际调用链和数据流，给出根因结论，更新 `02-analysis.md`

5.5 **打点失效升级**（逃生出口，日志分析后根因置信度仍为「模糊/未知」时触发）

   > 打点只能观测「代码层执行路径」，无法穿透平台层/宿主环境的静默拦截。根因仍不明确时，应立即转向网络搜索。

   **触发判定**：日志分析完成后根因置信度仍为「模糊/未知」，执行以下步骤：
   1. **升级搜索**：以「症状关键词 + 平台/框架版本 + 年份」搜索已知案例
   2. **输出搜索结论**：
      - 找到已知案例 → 更新根因假设，回到步骤 5 重新解读日志或设计新打点
      - 未找到 → 输出「已排除假设清单」，**暂停，等用户决定**（继续打点 / 寻求外部支持 / 带假设进入方案阶段）

6. **打点清理建议**（可选）：根因确认后，说明哪些打点可删除，哪些值得转为正式的可开关监控

**工具限制**：✅ Read/Grep 辅助确定打点位置；❌ Edit/Write（打点代码由用户手动添加，或经用户明确确认后 AI 添加）；❌ 未经用户确认不得自行运行复现步骤

## 输出字段

存在性验证结论、问题现象（复现/期望/实际）、代码定位（路径:行号+说明）、根因（直接+根本+调用链）、行业通病评估结论、影响范围（模块/平台/连带）、难度预判。

保存到 `.jira-fix/{JIRA-ID}/02-analysis.md`，更新 state.json。

## Red Flags — 禁止行为

- 跳过存在性验证，直接进入问题现象分析
- 存在性验证结论为「不存在/描述不符」但仍继续分析
- 以「Jira 描述已经很清楚」为由跳过代码定位，直接进入方案评估
- 以「这是常见 bug」为由跳过根因分析，直接复用历史方案
- **已知问题快搜触发条件满足，却以「先看代码」「先打点」为由跳过 WebSearch**
- 根因分析后跳过行业通病评估，直接进入影响范围
- 行业通病评估结论为「无可行解」但仍进入阶段3
- **多轮打点后根因仍不明确，继续增加打点而不触发 5.5 逃生出口**
- 分析阶段修改代码（本阶段严格只读）
