# Jira Bug 批量自动修复工作流 - 执行规则

> 本文档定义批量自动修复 Jira Bug 的管理流程
>
> **重要**：单个 bug 修复流程完全遵循 `jira-auto-fix-workflow` skill。Jira 数据获取统一通过 `jira-read --live` 完成。

## 触发词定义

**主要触发词**：`jira-auto-batch [Jira信息] [project1,project2,...]`

**Jira 信息格式**（AI 自动识别）：URL 列表、ID 列表或混合格式

**项目指定**（可选）：全局指定所有 bug 使用相同项目列表，未指定时每个 bug 独立自动识别

**与单个修复对比**：

| 特性 | jira-auto-fix | jira-auto-batch |
|------|---------------|-----------------|
| 处理数量 | 单个 bug | 批量 bug |
| 管理文档 | 无 | 自动创建日期文档 |
| 状态追踪 | 无 | 实时更新 |
| 失败处理 | 终止 | 记录后继续 |

---

## 批量管理流程

### 阶段0：初始化批量任务

**执行步骤**：
1. 解析用户输入（自动识别 URL/ID 格式，去重，按输入顺序排序）
2. 批量预获取 Jira 数据：调用 `jira-read --live ID1 ID2 ID3 ...`（一次性缓存所有 Jira 信息）
3. 创建管理文档：`jira-auto-batch-YYYY-MM-DD.md`（冲突时添加时间戳）
4. 初始化 Bug 清单表格（使用预获取的标题、优先级等信息）
5. 输出任务概览

**管理文档结构**：
- 任务概览（总数、完成数、状态统计）
- Bug 清单表格（#、ID、标题、优先级、项目、状态、分支、报告、备注）
- 修复详情（随修复进行添加）
- 批量修复总结（完成后生成）

### 阶段1-N：逐个修复 Bug

**核心原则**：每个 bug 独立调用 `jira-auto-fix-workflow` skill 的完整 7 阶段流程

**对每个 Bug 执行**：
1. 更新状态为 🔄 修复中，记录开始时间
2. 调用 jira-auto-fix 工作流（传递项目信息）
3. 更新状态：✅ 已完成（记录分支、报告、改动、耗时）或 ❌ 失败（记录原因）
4. 更新任务概览统计
5. 继续下一个 bug

**失败处理**：记录原因，标记失败状态，不阻断后续 bug 修复

**状态标识**：⏳ 待处理、🔄 修复中、✅ 已完成、❌ 失败

### 阶段 Final：生成批量修复总结

**执行步骤**：
1. 统计所有 bug 修复结果（成功/失败/未完成）
2. 计算成功率和总耗时
3. 列出成功、失败、未完成的 bug 清单及原因
4. 提供后续操作建议
5. 更新管理文档的"批量修复总结"部分

## 核心原则

### 1. 复用原则
- 100% 引用 `jira-auto-fix-workflow` skill
- 只负责管理层逻辑

### 2. 自动化原则
- 完全自动化，失败不阻断

### 3. 可追溯原则
- 完整修复记录，详细时间和状态

### 4. 安全原则
- 独立分支，支持中断恢复

---

## 管理文档规范

**文件命名**：`jira-auto-batch-YYYY-MM-DD.md`，冲突时添加 `-HHmmss`

**实时更新时机**：
1. 初始化后：创建文档和表格
2. 修复前/后：更新状态和详情
3. 完成后：生成总结

**Bug 清单表格**：#、Jira ID、标题、优先级、项目、状态、分支、报告、备注

---

## 输入格式

**灵活格式**（AI 自动识别）：
```
jira-auto-batch [backend,frontend]
YNOTR-12167
https://jira.mail.netease.com/browse/YNOTR-12168
YNOTR-12169, YNOTR-12170
```

支持：URL 列表、ID 列表、混合格式、带/不带项目指定

---

## 输出要求

- 语言：中文，Markdown 格式
- 管理文档：`jira-auto-batch-YYYY-MM-DD.md`（实时更新）
- 单个报告：`reports/[JIRA-ID]-analysis.md`

---

## 错误处理

- 解析失败：提示用户重新输入或确认
- 单个 bug 失败：记录原因，继续下一个
- 连续失败：建议检查环境配置
- 文档写入失败：重试，失败则警告

---

## 任务恢复

如果批量任务中断：
1. 读取现有管理文档
2. 识别未完成的 bug（状态为 ⏳ 或 🔄）
3. 询问用户是否继续从中断点恢复
4. 继续执行未完成的 bug
