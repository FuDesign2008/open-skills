# Jira Bug 修复工作流

> 默认自动模式，全流程无需确认。说「手动模式」或「手动修复」进入手动确认模式。

> **前提**：mcp-atlassian 已配置且 PAT 有效；Git 环境正常。
> **输出规范**：Markdown 结构化（标题/列表/表格/代码块），代码引用用 `startLine:endLine:filepath`；容易 → 精简，困难/极难 → 详细。
> **格式参考**：各阶段输出模板见 [reference.md](reference.md)。

## 触发词与模式

| 说法示例 | 模式 | 说明 |
|---------|------|------|
| 「修复这个 bug [URL]」「帮我修复 [URL]」`jira-fix [URL]` | 🤖 自动 | 默认，全流程自动执行 |
| 「手动修复 [URL]」「修复 [URL] 手动模式」`jira-fix [URL] --manual` | 👤 手动 | 方案/计划/提交需用户确认 |
| 「强制修复 [URL]」「跳过分级修复 [URL]」`jira-fix [URL] --force` | 🤖 自动 | 跳过难度分级，强制执行 |
| 「继续修复」「从上次继续」「恢复修复 [URL]」`jira-fix [URL] --resume` | 当前模式 | 从断点恢复 |

**模式识别规则**：触发词中含「手动」「我来确认」「一步一步」→ 手动模式；含「强制」「跳过分级」→ 跳过分级；含「继续」「恢复」「上次」→ 断点恢复；其余默认自动模式。

## 模式差异速查

| 阶段 | 🤖 自动模式 | 👤 手动模式 |
|------|------------|------------|
| 阶段0：前置检查 | P0 自动拦截，工作区自动 stash | P0 提示，工作区提示用户处理 |
| 阶段1：创建分支 | 多项目支持，自动 stash，自动切换已有分支 | 单项目，错误等待用户处理 |
| 阶段3.5：难度分级 | 极难 → 终止，输出标记报告 | 极难 → 列出风险，用户选择 A/B |
| 阶段4：评估方案 | AI 自动选择最佳方案 | 输出方案表，等用户选择 |
| 阶段5：制定计划 | 自动确认，直接进阶段6 | 等用户确认 |
| 阶段6：执行计划 | 执行 + Linter + 自动生成分析报告 | 执行 + Linter |
| 阶段7：Git 提交 | 自动 push + Jira 状态回写 | 生成命令 + 验证场景，等用户执行 |
| 中断恢复 | 自动恢复，不询问 | 询问是否恢复 |

|------|-----------|------|------|

| 阶段3：分析问题 | ❌ | ❌ | 只读，不修改任何文件 |
| 阶段4：评估方案 | ❌ | ❌ | 只读，不修改任何文件 |

| 阶段7：Git 提交 | ❌ | ✅（仅 git 命令） | 不改代码，只执行提交 |

## 通用原则

### 调查优先、证据驱动

1. **先调查再发言** — 对不确定的问题，先定位代码再给出判断；没有代码证据就没有发言权
2. **证据说话** — 以代码事实、报错信息说话，不凭猜测推断根因

### 主动提问

各阶段信息不足时，先提出 1～3 个关键问题，待补充后再继续。

**提问格式**：
```
1. 题干（一句话说清问题）
   - A 选项一
   - B 选项二
2. 下一题题干
   - A 选项一
   - B 选项二
```

**简答约定**：用户可直接回复「1A、2B」等形式，AI 按题号与选项字母解析后继续。

**一次一问**：若问题复杂或易产生歧义，改为逐个提问，降低认知负担。

## 状态持久化（中断恢复）

### 恢复触发
- 说「继续修复」「从上次继续」「恢复修复 [URL]」，或使用 `jira-fix [URL] --resume`
- 启动时自动检测已有进度：**[🤖 自动]** 自动恢复；**[👤 手动]** 提示是否恢复

### 状态目录
```
.jira-fix/{JIRA-ID}/
├── state.json           ← 进度跟踪（含 mode 字段）
├── 01-jira-info.md      ← 阶段2输出
├── 02-analysis.md       ← 阶段3输出
├── 02-grade.md          ← 阶段3.5难度分级结果
├── 03-options.md        ← 阶段4输出
├── 04-plan.md           ← 阶段5输出
├── 05-execution.md      ← 阶段6输出
└── 06-report.md         ← 阶段7输出
```

### state.json 结构
```json
{
  "jira_id": "YNOTR-12345",
  "jira_url": "https://your-jira.example.com/browse/YNOTR-12345",
  "mode": "auto",
  "current_phase": 3,
  "completed_phases": [0, 1, 2],
  "branch": "fix/jira-fix-YNOTR-12345",
  "grade": null,
  "selected_option": null,
  "started_at": "ISO_TIMESTAMP",
  "last_updated": "ISO_TIMESTAMP"
}
```

### 恢复逻辑
1. 检测到已有 state.json 时：
   - **[🤖 自动]** 自动读取，从 current_phase 继续，不询问
   - **[👤 手动]** 提示「发现未完成任务（阶段X），是否恢复？」
2. 恢复后从已保存的阶段输出文件继续，减少上下文依赖

### 清理
修复完成后保留 `.jira-fix/{JIRA-ID}/` 目录，state.json 设 `current_phase: "completed"`

## 阶段0：前置检查

> 任一检查失败则中断流程

### 执行步骤

1. **确定执行模式**：
   - 解析触发词，识别 `--manual` / `--force` / `--resume` 参数
   - 将 mode 写入 state.json：`"mode": "auto"` 或 `"mode": "manual"`

2. **mcp-atlassian 连通性检查**：
   调用 `jira_get_issue`(issue_key="{JIRA-ID}", fields="summary,priority") 仅获取标题和优先级
   - ✅ 成功：记录 Issue 标题和优先级，继续流程
   - ❌ 失败：**立即终止**，输出：
   ```
   ❌ mcp-atlassian 连接失败

   请检查：
   1. opencode.json 中是否配置了 mcp-atlassian
   2. JIRA_PERSONAL_TOKEN 是否有效
   3. JIRA_URL 是否正确（当前：https://your-jira.example.com）
   4. 网络是否可达
   ```

3. **P0 快速拦截**（仅自动模式）：
   - **[🤖 自动]** 若优先级为 P0 → 立即终止，提示：「P0 级别 bug 需要人工审核，请使用 --manual 模式」

4. **Git 环境检查**：
   - 是否在 Git 仓库中（不在则跳过阶段1）
   - **[🤖 自动]** 工作区不干净 → 自动 `git stash save "Auto-stash before jira-fix"`
   - **[👤 手动]** 工作区不干净 → 提示用户提交/暂存/放弃，等待确认

### 输出字段

Jira ID、Issue 标题、优先级、执行模式、mcp 状态、Git 状态。进入阶段1。

## 阶段1：创建 Git 修复分支

### 原则
- 从 `main_branch` 文件读取主开发分支，不存在则使用当前分支
- 统一命名：`fix/jira-fix-[JIRA-ID]`

### [🤖 自动模式] 多项目支持

**项目识别策略**（按优先级）：
1. 触发词中显式指定：`jira-fix [URL] project1,project2`
2. 自动扫描 workspace 根目录一/二级子目录（存在 `.git`）
3. 排除：`node_modules`、`.cursor`、`dist`、`build`、`.git`

**自动执行步骤**：
1. 检查 Git 状态，自动 stash 未提交修改
2. 自动识别基础分支（优先 release/x.x.x）
3. 自动提取 Jira ID 并创建分支
4. 分支已存在时自动切换

**自动错误处理**：
- 未提交修改：自动 `git stash save "Auto-stash before jira-fix"`
- 分支已存在：自动 `git checkout fix/jira-fix-[JIRA-ID]`
- 无法识别基础分支：使用当前分支，记录警告
- 不在 Git 仓库：报错终止

### [👤 手动模式] 单项目

**执行步骤**：
1. 检查 Git 状态和当前分支
2. 从 Jira URL 提取 ID（正则：`/browse/([A-Z]+-\d+)`）
3. 检查修复分支是否已存在，存在则中断提示用户
4. 读取 `main_branch` 文件（不存在则使用当前分支），创建修复分支

**错误处理**：
- 未提交修改：提示用户提交/暂存/放弃，等待确认
- 分支已存在：中断，提示用户手动处理
- 无法识别基础分支：列出所有分支，请用户指定

### 命名

`fix/jira-fix-[JIRA-ID]`，如 `fix/jira-fix-YNOTR-12167`

### 输出字段

Jira ID、基础分支、修复分支、状态；多项目时表格列出各项目。输出保存到 `.jira-fix/{JIRA-ID}/00-branch.md`

## 阶段2：读取 Jira 信息

### 原则
- 优先获取最新数据（`--live` 模式）
- API 不可用时读取本地缓存
- 复用 `jira-read` skill 的统一逻辑
- **工具限制**：✅ mcp-atlassian API / jira-read；❌ Edit/Write/Bash

### 执行步骤
1. 调用 `jira-read {JIRA-ID} --live`（直接从 API 获取最新数据）
2. jira-read 自动处理：
   - live 模式：调用 mcp-atlassian API 获取最新数据并更新缓存
   - 降级：API 不可用时读取本地缓存
   - 再降级：**[👤 手动]** 请用户手动提供；**[🤖 自动]** 报错终止
3. 提取结构化信息
4. 保存到 `.jira-fix/{JIRA-ID}/01-jira-info.md`，更新 state.json

### 提取信息清单

**必需**：Jira ID、标题、优先级（P0-P4）、状态

**详细**：问题描述、复现步骤、期望结果、实际结果、附件链接、评论历史

### 输出字段

Jira ID、标题、优先级、状态、数据来源、问题描述、复现步骤、期望/实际结果、评论摘要。

## 阶段3：分析问题（只读）

> 只读分析，不修改代码

### 原则
- 深度分析，追问「为什么」至少 3 次
- 结构化输出，中文描述
- **[👤 手动]** 信息不足时主动提问，格式见「通用原则 → 主动提问」
- **工具限制**：✅ Read/Grep/SemanticSearch；❌ Edit/Write/Bash

### 分析维度

**1. 问题现象**：Jira 信息、复现步骤、期望 vs 实际结果

**2. 代码定位**：使用 Grep/Glob/SemanticSearch 组合搜索关键代码

**3. 根因分析**：
- 数据流分析：用户操作→组件→函数→状态→渲染
- 调用链分析：追踪完整函数调用链
- 区分直接原因和根本原因

**4. 影响范围**：功能影响、平台影响（Web/Mobile）、模块影响、连带影响

**5. 难度预判**：
- 预估改动文件数
- 根因清晰度：清晰 / 基本清晰 / 模糊 / 未知
（供阶段3.5使用）

### 输出字段

问题现象（复现步骤/期望/实际）、代码定位（路径:行号+说明）、根因（直接+根本+调用链）、影响范围（模块/平台/连带）、难度预判（预估文件数、根因清晰度）。保存到 `.jira-fix/{JIRA-ID}/02-analysis.md`

### Red Flags — 禁止行为

- 以「Jira 描述已经很清楚」为由跳过代码定位，直接进入方案评估
- 以「这是常见 bug」为由跳过根因分析，直接复用历史方案
- 分析阶段修改代码（本阶段严格只读，违反阶段工具约束）
- **[🤖 自动]** 所有方案风险 > 中时不暂停，直接执行

## 阶段3.5：难度分级 + 模式决策网关

> 基于阶段3的分析结果，使用规则制判定难度等级，决定后续行为

### 🔴 极难触发条件（满足任一即为极难）

| 条件 | 判断依据 |
|------|---------|
| 根因未知 | 分析后仍无法定位根本原因（知道大概模块但无法定位到具体逻辑） |
| 涉及架构变更 | 需要重构模块间通信、变更设计模式、调整核心数据结构 |
| 涉及数据迁移 | 需要数据库 schema 变更或数据格式迁移 |
| API 协议变更 | 需要修改对外接口签名，影响调用方 |
| 改动范围过大 | 预估改动文件 >10 个 或 预估行数变更 >500 行 |
| 跨仓库/跨服务 | 需要在多个独立 Git 仓库中协同修改 |

### 等级判定（未命中极难条件时）

- 预估文件 ≤ 3 个，根因**清晰** → 🟢 **容易**
- 预估文件 4–10 个，根因**基本清晰** → 🟡 **中等**
- 预估文件 ≤ 10 个，根因**较清晰**但改动较多 → 🟠 **困难**

### 模式 × 等级 → 行为

| 等级 | 🤖 自动模式 | 👤 手动模式 |
|------|------------|------------|
| 🟢 容易 | ✅ 正常执行 | 💡 提示「此问题较简单，可切换自动模式」，继续手动流程 |
| 🟡 中等 | ✅ 正常执行 | 💡 提示「此问题风险可控，可切换自动模式」，继续手动流程 |
| 🟠 困难 | ✅ 执行，阶段6完成后暂停等用户审查代码再提交 | ✅ 正常手动流程 |
| 🔴 极难 | ❌ **终止**，输出标记报告 | ⚠️ **风险提示**，用户选择 A/B |

### 输出字段

难度等级（🟢/🟡/🟠/🔴）、命中条件列表、行为提示（终止报告/风险选项 A-B/建议切换模式）、后续行动建议。分级结果保存到 `.jira-fix/{JIRA-ID}/02-grade.md`，更新 state.json `"grade"` 字段

## 阶段4：评估方案

### [🤖 自动模式] 自动选择

自动生成 2-3 个方案，按以下优先级选择：
1. 更彻底解决问题（而非临时修复）
2. 符合代码规范和最佳实践
3. 改善代码质量（重构优于补丁）
4. 复杂度相近时选改动较少的

```
if (所有方案风险 > 中) {
  → 继续执行，但阶段5后强制等用户确认
} else {
  → 选择推荐度最高的方案
  → 推荐度相同时选复杂度最低的
}
```

### [👤 手动模式] 等用户选择

若缺少选型偏好，先提出 1～2 个关键问题，再输出方案对比与推荐。

### [👤 手动模式] 方案迭代（可多轮）

用户说「修改方案」「完善方案」「优化方案」时，在上一轮基础上重新输出，无需重新生成全部方案：

- **修改方案** — 调整思路、修正偏差、规避识别出的风险
- **完善方案** — 补充遗漏细节、明确边界条件
- **优化方案** — 提升效率、降低复杂度

### 每个方案必须包含

1. 核心思路（1-2 句话）
2. 需要修改的文件/模块
3. 优点（≥2）、缺点（≥1）
4. 复杂度（低/中/高）、风险等级（低/中/高）
5. 适用场景

### 输出字段

方案对比表（核心思路/复杂度/风险/推荐度）+ 各方案详情 + 选定/推荐方案及理由。[🤖 自动] 自动选择后进入阶段5；[👤 手动] 等用户选择。保存到 `.jira-fix/{JIRA-ID}/03-options.md`

### Red Flags — 禁止行为

- 只生成 1 个方案，以「方向已明确」为由跳过方案对比
- **[🤖 自动]** 所有方案风险 > 中时仍自动选择，不暂停等确认
- **[👤 手动]** 用户未选方案时自行推进到阶段5
- 在方案评估阶段修改代码（违反只读约束）

---

## 阶段5：制定计划

> 详细可执行的修改计划

### 计划必须包含 8 个部分

1. 问题根因回顾（阶段3）
2. 选定方案回顾（阶段4）
3. 架构设计（可选 Mermaid 图）
4. 文件修改清单（表格：文件、内容、类型）
5. 修改顺序（考虑依赖关系）
6. 测试场景（Web端/Mobile端/兼容性）
7. 影响范围（服务层/Web/Mobile/风险等级）
8. 回滚方案

### 确认方式

| 场景 | 行为 |
|------|------|
| **[🤖 自动]** 普通情况 | 自动确认，立即进入阶段6 |
| **[🤖 自动]** 难度🟠困难 或 方案风险>中 | 输出计划后暂停，等用户确认后继续 |
| **[👤 手动]** 普通情况 | 等用户确认 |
| **[👤 手动]** 极难选B | 强制要求用户二次确认「我已知晓风险，继续执行」后才进阶段6 |

保存到 `.jira-fix/{JIRA-ID}/04-plan.md`

### Red Flags — 禁止行为

- 制定计划阶段修改代码（违反只读约束，见「阶段工具约束」表）
- **[🤖 自动]** 困难等级或高风险方案未暂停，直接跳入阶段6
- 计划缺少测试场景或回滚方案就推进到阶段6
- **[👤 手动]** 未等用户确认计划就进入执行

---

## 阶段6：执行计划

> 严格按计划执行，逐步更新 TODO 状态

### 前置检查

确认当前在正确修复分支（`fix/jira-fix-[JIRA-ID]`），工作区干净

### TODO 管理

使用 `TodoWrite` 创建和更新任务状态（pending/in_progress/completed）

### 代码质量要求

1. 函数职责单一
2. 命名清晰（函数用动词，变量用名词）
3. 避免概念混淆
4. 向后兼容

### 代码质量检查

1. **Linter 检查**：修改后用 `ReadLints` 检查，新引入的错误必须修复
2. **TypeScript 检查**（自动检测）：项目存在 `tsconfig.json` 时自动调用 `typescript-check` skill，新引入类型错误必须修复

### [🤖 自动模式] 额外能力

**多项目代码修改**：
- 根据文件路径前缀定位对应项目
- 每个项目独立检查 Linter
- 记录每个项目的修改文件列表和行数统计（为阶段7分项目提交准备）

**报告生成**：
执行完成后自动生成 `reports/[JIRA-ID]-analysis.md`，包含：
基本信息、问题分析、根因、解决方案、修改文件清单、遇到的问题、测试建议

### 执行后行为

| 场景 | 行为 |
|------|------|
| **[🤖 自动]** 普通情况 | 自动进入阶段7 |
| **[🤖 自动]** 难度🟠困难 | 暂停，输出执行摘要，等用户审查代码后确认继续 |
| **[👤 手动]** 普通情况 | 等用户确认后进入阶段7 |
| **[👤 手动]** 极难选B | 暂停，等用户审查代码，不自动提交 |

### 执行报告

每次执行完成后必须输出（自动模式简化版，手动模式完整版）：

- 已修改文件列表及关键改动说明
- 是否有偏离计划的调整（如有，说明原因）
- 验证要点 / 自查清单（供手动模式用户代码审查参考）
- **[🤖 自动]** 执行顺利 → 自动进入阶段7；遇到计划外问题 → 先暂停确认

保存到 `.jira-fix/{JIRA-ID}/05-execution.md`

### Red Flags — 禁止行为

- 跳过 Linter 检查，以「只是逻辑改动」为由认为不必要
- 偏离计划修改了计划外文件，未说明原因
- **[🤖 自动]** 遇到计划外问题未暂停，自行决策继续
- 执行完成未输出执行报告直接进入阶段7

## 阶段7：Git 提交代码

### [🤖 自动模式] 自动提交

**执行步骤**：
1. 收集提交信息（Jira ID、根因、修复方案、修改文件、报告路径）
2. 调用 `git-commit` SKILL（execute=true，自动模式）：自动 add / commit / push
3. **Jira 状态自动回写**：
   - 调用 `jira_get_transitions`(issue_key="{JIRA-ID}") 获取可用状态
   - 执行状态转换：优先匹配 "Code Review" > "In Review" > "In Progress"
   - 无匹配时跳过，记录警告
   - 调用 `jira_add_comment`(issue_key="{JIRA-ID}", body=修复评论)

**修复评论字段**：修复分支、Commit、根因、修复方案、修改文件、分析报告路径。

### [👤 手动模式] 生成命令

**执行步骤**：
1. 收集提交信息（Jira ID、问题描述、根本原因、修复方案、修改文件）
2. 调用 `git-commit` SKILL 生成 commit message 和命令（不自动执行）
3. 生成验证场景（功能场景 ≥2、边界场景 ≥2、回归场景 ≥2）
4. 等用户确认后执行命令

**验证场景**：功能场景 ≥2、边界场景 ≥2、回归场景 ≥2（含操作步骤和预期结果）。修复评论同自动模式，额外包含验证场景章节。

### Commit Message 格式

```
<type>(<scope>): <subject> <Jira-ID>
```

示例：`fix(ai-summary): 修复分享链接中AI摘要按钮显示问题 YNOTR-12167`

**Type 类型**：fix、feat、refactor、perf、style、docs、test

**Scope 示例**：ai-summary、share、auth、api、ui、core

### 完成输出字段

修复分支、Commit Hash、推送状态、Jira 回写状态、分析报告路径；多项目时用表格。**错误处理**：回写失败不阻断流程，仅输出警告并记录到报告。

保存到 `.jira-fix/{JIRA-ID}/06-report.md`

### Red Flags — 禁止行为

- **[👤 手动]** 未等用户确认直接执行 git push
- Jira 回写失败后未输出警告，静默跳过
- Commit message 遗漏 Jira ID
- 多项目场景只提交了部分项目

## 安全机制（自动模式）

1. 修改前自动 stash：`git stash save "Auto-stash before jira-fix"`
2. **改动阈值检测**：
   - 警告：修改 >10 个文件 或 >500 行
   - 阻断：修改 >20 个文件 或 >1000 行（需 `--force` 解除）
3. Linter 错误自动阻断流程
4. 详细操作日志，记录所有自动决策

## 常见错误

| 错误 | 后果 | 修正 |
|------|------|------|
| 跳过代码定位直接给方案 | 方案治标不治本 | 返回阶段3完成根因定位 |
| 自动模式遇极难仍继续执行 | 高风险改动失控 | 触发阶段3.5网关，终止流程 |
| 手动模式跳过方案确认直接制定计划 | 执行错误方案 | 等用户选择后再进阶段5 |
| 执行完成未做 Linter 检查 | 引入新的代码错误 | ReadLints 是必须步骤，不可跳过 |
| 分析阶段使用 Edit/Write 修改代码 | 破坏只读约束 | 见「阶段工具约束」表 |

## 批量修复

批量修复需要借助**任务编排工具**——即能将一组任务拆解为独立子任务、逐个执行并跟踪状态的工具。不同平台的命令名称可能不同（如 `/ralph`、`/autopilot`、`/ultrawork` 等），使用时以当前平台支持的编排工具为准。

### 核心思路

将每个 Jira bug 描述为一个独立任务，交由编排工具依次调用 `jira-fix [URL]` 完成修复。编排工具负责：
- 逐个执行，单个失败不阻断后续
- 跟踪整体进度，支持断点恢复
- 完成后汇总结果

### 示例（以 /ralph 为例）

```
/ralph 批量修复以下 Jira bug：
YNOTR-12167
YNOTR-12168
https://your-jira.example.com/browse/YNOTR-12169
```
