# Jira Bug 修复工作流 - 执行规则

> 本文档定义了执行 Jira Bug 修复时的核心规则。

## 触发词定义

**主要触发词**：`jira-fix [Jira URL]` 或 `修复 bug：[Jira URL]`

**单阶段触发词**：

| 触发词 | 阶段 | 说明 |
|--------|------|------|
| `创建分支` | 阶段1 | 创建修复分支 |
| `读取jira` | 阶段2 | 读取Jira信息 |
| `分析问题` | 阶段3 | 问题分析 |
| `评估方案` | 阶段4 | 评估解决方案 |
| `制定计划` | 阶段5 | 制定修复计划 |
| `执行计划` | 阶段6 | 执行代码修改 |
| `提交代码` | 阶段7 | Git提交建议 |

**触发响应**：确认触发 → 说明阶段 → 读取Jira → 创建分支 → 执行流程

## 状态持久化（中断恢复）

### 触发词
- `jira-fix {JIRA-ID} --resume` - 从断点恢复
- `jira-fix {URL}` - 正常启动（如有未完成状态，提示是否恢复）

### 状态目录
```
.jira-fix/{JIRA-ID}/
├── state.json           ← 进度跟踪
├── 01-jira-info.md      ← 阶段 2 输出
├── 02-analysis.md       ← 阶段 3 输出
├── 03-options.md        ← 阶段 4 输出
├── 04-plan.md           ← 阶段 5 输出
├── 05-execution.md      ← 阶段 6 输出
└── 06-report.md         ← 阶段 7 输出
```

### state.json 结构
```json
{
  "jira_id": "YNOTR-12345",
  "jira_url": "https://jira.mail.netease.com/browse/YNOTR-12345",
  "current_phase": 3,
  "completed_phases": [1, 2],
  "branch": "fuyg-jira-fix-YNOTR-12345",
  "selected_option": null,
  "started_at": "ISO_TIMESTAMP",
  "last_updated": "ISO_TIMESTAMP"
}
```

### 恢复逻辑
1. `--resume` 或检测到已有 state.json 时：
   - 读取 state.json
   - 读取已完成阶段的输出文件
   - 从 current_phase 继续执行
2. 正常启动但 state.json 已存在时：
   - 提示"发现未完成的修复任务（阶段 X），是否恢复？"
   - 用户选择恢复或重新开始

### 各阶段输出规则
每个阶段完成时：
1. 将输出写入对应文件（如 `02-analysis.md`）
2. 更新 state.json 的 current_phase 和 completed_phases
3. 下一阶段读取前置阶段的输出文件（减少上下文依赖）

### 清理
修复完成（阶段 7 成功）后：
- 保留 .jira-fix/{JIRA-ID}/ 目录作为历史记录
- 将 state.json 中 current_phase 设为 "completed"

## 阶段0：前置检查

> 在创建 Git 分支之前，验证关键依赖是否可用。任一检查失败则中断流程。

### 执行步骤

1. **mcp-atlassian 连通性检查**：
   调用 `jira_get_issue`(issue_key="{JIRA-ID}", fields="summary") 仅获取标题
   - ✅ 成功：记录 Issue 标题，输出确认信息，继续流程
   - ❌ 失败：**立即中断**，输出以下错误并终止：

```
❌ mcp-atlassian 连接失败

请检查：
1. opencode.json 中是否配置了 mcp-atlassian
2. JIRA_PERSONAL_TOKEN 是否有效
3. JIRA_URL 是否正确（当前：https://jira.mail.netease.com）
4. 网络是否可达

配置参考：
{
  "mcp-atlassian": {
    "command": ["uvx", "mcp-atlassian"],
    "environment": {
      "JIRA_URL": "https://jira.mail.netease.com",
      "JIRA_PERSONAL_TOKEN": "your_pat_here",
      "JIRA_SSL_VERIFY": "false"
    }
  }
}
```

2. **Git 环境检查**：
   - 是否在 Git 仓库中（不在则提示并跳过阶段 1）
   - 工作区是否干净（不干净则提示处理）

### 输出格式

```
## 阶段0完成：前置检查通过

**Jira ID**: YNOTR-12167
**Issue 标题**: [标题]（已确认存在）
**mcp-atlassian**: ✅ 已连接
**Git 仓库**: ✅ / ⚠️ 不在 Git 仓库中

---
进入阶段1：创建 Git 修复分支
```

## 阶段1：创建 Git 修复分支

### 原则
- 从 Git 仓库根目录的 `main_branch` 文件读取主开发分支，不存在则使用当前分支
- 统一命名：`fuyg-jira-fix-[JIRA-ID]`
- 确保工作区干净（无未提交修改）

### 执行步骤
1. 检查Git状态和当前分支
2. 从Jira URL提取ID（正则：`/browse/([A-Z]+-\d+)`）
3. 检查修复分支 `fuyg-jira-fix-[JIRA-ID]` 是否已存在，如果存在则中断并提示用户
4. 确定基础分支并创建修复分支：
   - 如果 `main_branch` 文件存在：读取第一行（去除首尾空白）作为基础分支，切换到该分支，然后创建修复分支 `git checkout -b fuyg-jira-fix-[JIRA-ID]`
   - 如果 `main_branch` 文件不存在：直接基于当前分支创建修复分支 `git checkout -b fuyg-jira-fix-[JIRA-ID]`

### 命名规范

| Jira URL示例 | Jira ID | 分支名称 |
|-------------|---------|---------|
| `.../browse/YNOTR-12167` | YNOTR-12167 | fuyg-jira-fix-YNOTR-12167 |
| `.../browse/PROJ-123` | PROJ-123 | fuyg-jira-fix-PROJ-123 |

要求：前缀`fuyg-jira-fix-` + Jira ID（原样保留大小写）+ 连字符分隔

### 错误处理

**未提交修改**：提示用户提交、暂存或放弃，等待确认后继续

**修复分支已存在**：中断流程，提示用户分支 `fuyg-jira-fix-[JIRA-ID]` 已存在，请手动处理

**main_branch 文件读取失败或内容无效**：使用当前分支

**main_branch 中指定的分支不存在**：使用当前分支

**无法识别基础分支**：列出所有分支，请用户指定

**Git操作失败**：显示错误信息，提供降级方案（手动创建分支）

**不在Git仓库**：提示检查目录，跳过阶段1进入阶段2

### 输出格式

```
## 阶段1完成：Git分支已创建

**Jira ID**: YNOTR-12167
**基础分支**: release/8.2.30
**基础分支来源**: main_branch 文件（如不存在则使用当前分支）
**修复分支**: fuyg-jira-fix-YNOTR-12167
**状态**: 已切换到修复分支

---
准备进入阶段2：读取 Jira 信息
```

## 阶段2：读取 Jira 信息

### 原则
- 优先获取最新数据并更新缓存
- API 不可用时读取本地缓存
- 复用 `jira-read` skill 的统一逻辑（包含 live + 降级）

### 执行步骤
1. 调用 `jira-read {JIRA-ID} --live`（优先获取最新数据）
2. jira-read 自动处理：
   - --live 模式：直接调用 mcp-atlassian API 获取最新数据并更新缓存
   - 降级：API 不可用时读取本地缓存
   - 再降级：API 和本地均不可用时，请用户手动提供 Jira 信息
3. 获取结构化 Jira 信息
4. 将阶段 2 输出保存到 `.jira-fix/{JIRA-ID}/01-jira-info.md`
5. 更新 state.json

### 提取信息清单

**必需**：Jira ID、标题、优先级（P0-P4）、状态

**详细**：问题描述、复现步骤、期望结果、实际结果、附件链接、评论历史

### 降级策略

mcp-atlassian API 不可用且本地无缓存时，请用户手动提供 Jira 信息

### 输出格式

```
## 阶段2完成：Jira信息已读取

**Jira ID**: YNOTR-12167
**标题**: [标题]
**优先级**: P1
**状态**: 待处理
**数据来源**: 本地缓存 / 自动下载

**问题描述**: [描述]
**复现步骤**: 1. ... 2. ...
**期望结果**: [期望]
**实际结果**: [实际]

**评论摘要**: 共 X 条评论
> 最新评论 - [用户] ([时间]): [内容摘要]...

---
准备进入阶段3：分析问题
```

输出同时保存到 .jira-fix/{JIRA-ID}/01-jira-info.md

---

## 阶段3：分析问题（只读模式）

本阶段原则与步骤参见 problem-solving-workflow 阶段 2（分析问题）；以下仅列出与 Jira Bug 修复流程的差异与输出格式。

### 原则
- 只读分析，不修改代码
- 深度分析，追问"为什么"至少3次
- 结构化输出，中文描述
- 信息不足时主动提出 1～3 个关键问题（如复现步骤、环境、期望行为），待用户补充后再继续分析（与 problem-solving-workflow 一致）。向用户提问时采用其「提问与简答约定」，用户可回复如 1A、2B。

### 信息补全（如需要）

若缺少分析所需关键信息（除 Jira 已提供外，如环境/版本、已做尝试等），先列出 1～3 个关键问题并请用户补充，待回复后再执行下述分析维度。

### 分析维度

**1. 问题现象**：Jira信息、复现步骤、期望vs实际结果

**2. 代码定位**：使用Grep/Glob/SemanticSearch组合搜索关键代码

**3. 根因分析**：
- 数据流分析：用户操作→组件→函数→状态→渲染
- 调用链分析：追踪完整函数调用链
- 区分直接原因和根本原因

**4. 影响范围**：功能影响、平台影响（Web/Mobile）、用户影响、技术影响、连带影响

### 输出格式

```
## 阶段3：分析问题

### 1. 问题现象
**Jira ID**: [ID]
**标题**: [标题]
**复现步骤**: 1. ... 2. ...
**期望结果**: [期望]
**实际结果**: [实际]

### 2. 相关代码定位
**核心文件**:
- `path/file.js:行号` - [说明]

**关键函数/类**:
- `functionName()` - [作用]

### 3. 根因分析
**数据流**: [流程]
**调用链**: [调用关系]
**直接原因**: [直接导致问题的逻辑]
**根本原因**: [为什么会有这样的逻辑]

### 4. 影响范围
**受影响模块**: [列表]
**平台**: Web端/Mobile端
**潜在连带影响**: [列表]

### 5. 风险点
- [风险1]
- [风险2]

---
准备进入阶段4：评估方案
```

输出同时保存到 .jira-fix/{JIRA-ID}/02-analysis.md

---

## 阶段4：评估方案（交互模式）

本阶段原则与步骤参见 problem-solving-workflow 阶段 3（评估方案）；以下仅列出与 Jira 流程的差异与输出格式。

### 原则
- 至少2个方案，最多5个（根据复杂度）
- 客观对比优缺点
- 推荐但不强制，等待用户决策
- 若缺少选型偏好、优先级（如更看重改动范围还是实施速度）等，先向用户提出 1～2 个关键问题，再输出方案对比与推荐（与 problem-solving-workflow 一致）。提问格式与简答方式见 problem-solving-workflow「提问与简答约定」，用户可回复如 1A、2B。

### 每个方案必须包含
1. 核心思路（1-2句话）
2. 需要修改的文件/模块
3. 实施步骤（简要）
4. 优点（至少2个）
5. 缺点（至少1个）
6. 复杂度（低/中/高）
7. 风险等级（低/中/高）
8. 适用场景

### 输出格式

```
## 阶段4：评估方案

| 方案 | 核心思路 | 优点 | 缺点 | 复杂度 | 风险 | 推荐度 |
|------|---------|------|------|--------|------|--------|
| 方案1 | [思路] | [列举] | [列举] | 低/中/高 | 低/中/高 | ⭐⭐⭐⭐⭐ |
| 方案2 | [思路] | [列举] | [列举] | 低/中/高 | 低/中/高 | ⭐⭐⭐⭐ |

[每个方案的详细说明]

### 推荐方案
推荐：方案X
理由：1. ... 2. ... 3. ...

---
请选择方案编号，进入阶段5：制定计划
```

输出同时保存到 .jira-fix/{JIRA-ID}/03-options.md

## 阶段5：制定计划（Plan模式）

本阶段原则与步骤参见 problem-solving-workflow 阶段 4（制定计划）；以下仅列出与 Jira 流程的差异与输出格式。

### 原则
- 详细具体，每步明确
- 考虑依赖关系和执行顺序
- 包含测试场景
- 可选使用Mermaid图表
- 若计划范围、优先级或约束（如必须先改某模块）不明确，先向用户提出 1～2 个关键问题，再输出完整修改计划（与 problem-solving-workflow 一致）。提问格式与简答方式见 problem-solving-workflow「提问与简答约定」，用户可回复如 1A、2B。

### 计划必须包含8个部分

1. 问题根因回顾（阶段3）
2. 选定方案回顾（阶段4）
3. 架构设计（可选Mermaid图）
4. 文件修改清单（表格：文件、内容、类型）
5. 修改顺序（考虑依赖）
6. 测试场景（Web端/Mobile端/兼容性）
7. 影响范围（服务层/Web/Mobile/风险等级）
8. 回滚方案

### 等待确认

计划创建后说明"计划已创建，请确认"，用户确认后进入阶段6

### 输出格式

输出同时保存到 .jira-fix/{JIRA-ID}/04-plan.md

---

## 阶段6：执行计划（代码修改）

本阶段原则与步骤参见 problem-solving-workflow 阶段 5（执行计划）；以下仅列出与 Jira 流程的差异与输出格式。

### 原则
- 严格按计划执行
- 逐步进行，更新TODO状态
- 检查Linter质量
- 只做代码修改，不包含Git提交
- 若执行中出现计划未覆盖的歧义或边界情况，先向用户确认再动手修改，避免擅自假设（与 problem-solving-workflow 一致）。

### 前置检查

确认当前在正确的修复分支（`fuyg-jira-fix-[JIRA-ID]`）、工作区干净

### TODO管理

使用`TodoWrite`创建和更新任务状态（pending/in_progress/completed）

### 代码质量要求

1. 函数职责单一
2. 命名清晰（函数用动词，变量用名词）
3. 避免概念混淆
4. 向后兼容

### 代码质量检查

1. **Linter 检查**：修改后用 `ReadLints` 检查，新引入的错误必须修复
2. **TypeScript 检查**（可选）：如果项目是 TypeScript 项目，调用 `typescript-check` skill 进行类型检查
   - 触发词："类型检查" 或 "type-check"
   - 新引入的类型错误必须修复

### 输出格式

输出同时保存到 .jira-fix/{JIRA-ID}/05-execution.md

## 阶段7：Git 提交代码

### 原则
- 调用 `git-commit` SKILL 生成 commit message 和命令
- 不自动执行，由用户决定何时提交
- 强调测试优先

### 执行步骤

1. **收集提交信息**：
   - Jira ID: [从阶段2获取]
   - 问题描述: [从阶段3获取]
   - 根本原因: [从阶段3获取]
   - 修复方案: [从阶段5获取]

   - 修改文件: [从阶段6获取]
   - 分支名称: fuyg-jira-fix-[JIRA-ID]

2. **调用 git-commit SKILL**：
   ```
   调用 git-commit SKILL，传递以下信息：
   - type: "fix"（根据实际情况确定）
   - scope: [根据修改模块确定，如 ai-summary、share、auth、api、ui、core]
   - subject: [从Jira标题或问题描述中提取，不超过50字符]
   - jira_id: [JIRA-ID]
   - problem_description: [问题描述]
   - root_cause: [根本原因]
   - solution: [修复方案]
   - modified_files: [修改文件列表]
   - branch: fuyg-jira-fix-[JIRA-ID]
   ```

3. **等待用户确认后执行命令**

### 7.2 生成验证场景

Git 提交完成后、写入 Jira 评论前，**基于前序阶段的修复上下文动态生成验证场景**：

**信息来源**：
- 阶段3（问题现象、复现步骤、影响范围）
- 阶段6（修改文件、修改逻辑）

**生成规则**：

| 类型 | 数量要求 | 覆盖重点 |
|------|---------|---------|
| **功能场景（正向）** | ≥ 2 条 | 直接覆盖 bug 修复的核心路径（原报告中的复现步骤 → 期望结果） |
| **边界场景** | ≥ 2 条 | 异常输入、边缘状态、特殊条件（如空值、已有标记、嵌套结构等） |
| **回归场景** | ≥ 2 条 | 与修改模块相邻的关联功能，验证未受影响（原有功能保持正常） |

**每条场景格式**：
```
[序号]. [场景名称]
   操作：[具体步骤，1-3步]
   预期：[用户可观察到的结果]
```

生成的场景写入下方 Jira 修复评论模板的 `验证场景` 区块。

### 7.3 Jira 状态回写（可选）

Git 提交完成后，自动执行 Jira 状态更新：

1. **获取可用状态转换**：调用 `jira_get_transitions`(issue_key="{JIRA-ID}")
2. **执行状态转换**：调用 `jira_transition_issue`(issue_key="{JIRA-ID}", transition="Code Review")
   - 状态名称因项目/工作流而异，优先匹配："Code Review" > "In Review" > "In Progress"
   - 如果无匹配状态，跳过转换并输出警告
3. **添加修复评论**：调用 `jira_add_comment`(issue_key="{JIRA-ID}", body=修复评论)

**修复评论模板**：
```
**AI 辅助修复报告**

- **修复分支**: fuyg-jira-fix-{JIRA-ID}
- **Commit**: {commit_hash}
```

### Commit Message 格式说明

`git-commit` SKILL 会按照以下格式生成 commit message：

```
<type>(<scope>): <subject> <Jira-ID>
```

**Type类型**：fix、feat、refactor、perf、style、docs、test

**Scope示例**：ai-summary、share、auth、api、ui、core

**Subject**：中文简要描述（不超过50字符）

**说明**：详细参数（问题描述、根本原因、修复方案等）仅用于上下文分析，不写入 commit message。

### 完整示例

```
fix(ai-summary): 修复分享链接中AI摘要按钮显示问题 YNOTR-12167
```

**注意**：详细参数（问题描述、根本原因、修复方案、修改文件）仍会收集并传递给 `git-commit` SKILL，但仅用于上下文分析，不会写入 commit message。

## 核心原则

### 1. 问题定位
- 深度优先：追问"为什么"至少3次，找到根本原因
- 全面分析：完整数据流、调用链、影响范围
- 证据驱动：基于代码和测试，不凭猜测

### 2. 方案评估
- 多样化：至少2个方案，包含不同类型
- 客观性：优缺点都说明，不隐藏风险
- 权衡考虑：复杂度vs效果、短期vs长期、局部vs全局

### 3. 代码修改
- 职责单一：一个函数一个职责
- 命名清晰：见名知意，动词+名词
- 概念分离：区分相似概念（如"支持"vs"显示"）
- 向后兼容：保留旧接口，考虑旧数据

### 4. 质量保证
- Linter零容忍：新引入的错误必须修复
- 多端一致：Web和Mobile同步修改
- 提交规范：生成完整commit message和命令
- 测试建议：功能、兼容性、回归测试
