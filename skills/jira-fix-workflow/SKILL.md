# Jira Bug 修复工作流

> 默认自动模式，全流程无需确认。说「手动模式」或「手动修复」进入手动确认模式。
>
> **前提**：mcp-atlassian 已配置且 PAT 有效；Git 环境正常。
> **输出规范**：Markdown 结构化（标题/列表/表格/代码块），代码引用用 `startLine:endLine:filepath`；容易 → 精简，困难/极难 → 详细。
> **格式参考**：各阶段输出模板见 [reference.md](reference.md)。

## 触发词与模式

| 说法示例 | 模式 | 说明 |
|---------|------|------|
| 「修复这个 bug [URL]」「帮我修复 [URL]」`jira-fix [URL]` | 🤖 自动 | 默认，全流程自动执行 |
| 「手动修复 [URL]」「修复 [URL] 手动模式」`jira-fix [URL] --manual` | 👤 手动 | 方案/计划/提交需用户确认 |
| 「强制修复 [URL]」「跳过分级修复 [URL]」`jira-fix [URL] --force` | 🤖 自动 | 跳过难度分级，强制执行 |
| 「继续修复」「从上次继续」「恢复修复 [URL]」`jira-fix [URL] --resume` | 当前模式 | 从断点恢复 |

**模式识别规则**：触发词中含「手动」「我来确认」「一步一步」→ 手动模式；含「强制」「跳过分级」→ 跳过分级；含「继续」「恢复」「上次」→ 断点恢复；其余默认自动模式。

## 模式差异速查

| 阶段 | 🤖 自动模式 | 👤 手动模式 |
|------|------------|------------|
| 阶段0：前置检查 | P0 自动拦截，工作区自动 stash | P0 提示，工作区提示用户处理 |
| 阶段1：创建分支 | 多项目支持，自动 stash，自动切换已有分支 | 单项目，错误等待用户处理 |
| 阶段3.5：难度分级 | 极难 → 终止，输出标记报告 | 极难 → 列出风险，用户选择 A/B |
| 阶段4：探索与审查方案 | AI 自动探索并审查，风险>中则暂停 | 输出方案表→用户选择→审查→确认后进阶段5 |
| 阶段5：制定计划 | 自动确认，直接进阶段6 | 等用户确认 |
| 阶段6：执行计划 | 执行 + Linter + 自动生成分析报告 | 执行 + Linter |
| 阶段7：提交与回顾 | 自动 push + Jira 状态回写 | 生成命令 + 验证场景，等用户执行 |
| 中断恢复 | 自动恢复，不询问 | 询问是否恢复 |

## 阶段工具约束

| 阶段 | Edit/Write | Bash | 说明 |
|------|-----------|------|------|
| 阶段2：读取 Jira | ❌ | ❌ | 只调用 API，不修改任何文件 |
| 阶段3：分析问题 | ❌ | ❌ | 只读，不修改任何文件 |
| 阶段4：探索与审查方案 | ❌ | ❌ | 只读，不修改任何文件 |
| 阶段5：制定计划 | ❌ | ❌ | 只输出计划文本 |
| 阶段6：执行计划 | ✅ | ✅ | 允许代码修改和命令执行 |
| 阶段7：提交与回顾 | ❌ | ✅（仅 git 命令） | 不改代码，只执行提交 |

## 通用原则

- **先调查再发言**：没有代码证据不做判断，以事实而非猜测说话
- **主动提问**：信息不足时提 1～3 个选择题；用户可「1A2B」形式简答；复杂问题一次一问
- **Jira 状态边界**：研发角色只能将 issue 流转到「已修复」，**禁止流转到「关闭」「验证通过」**（这些状态由 QA 操作）

## 状态持久化（中断恢复）

**恢复**：检测到 state.json 时，**[🤖 自动]** 直接从 `current_phase` 继续；**[👤 手动]** 询问是否恢复。**清理**：完成后 state.json 设 `current_phase: "completed"`。

### 状态目录

```
.jira-fix/{JIRA-ID}/
├── state.json           ← 进度跟踪（含 mode 字段）
├── 01-jira-info.md      ← 阶段2输出
├── 02-analysis.md       ← 阶段3输出
├── 02-grade.md          ← 阶段3.5难度分级结果
├── 03-options.md        ← 阶段4输出
├── 04-plan.md           ← 阶段5输出
├── 05-execution.md      ← 阶段6输出
└── 06-report.md         ← 阶段7输出
```

### state.json 结构

```json
{
  "jira_id": "YNOTR-12345",
  "jira_url": "https://your-jira.example.com/browse/YNOTR-12345",
  "mode": "auto",
  "current_phase": 3,
  "completed_phases": [0, 1, 2],
  "branch": "fix/jira-fix-YNOTR-12345",
  "grade": null,
  "selected_option": null,
  "started_at": "ISO_TIMESTAMP",
  "last_updated": "ISO_TIMESTAMP"
}
```

## 阶段0：前置检查

> 任一检查失败则中断流程

### 执行步骤

1. **识别模式**：解析触发词，识别 `--manual` / `--force` / `--resume`，写入 state.json
2. **mcp 连通检查**：调用 `jira_get_issue`（仅获取标题/优先级）；失败立即终止，提示检查配置/PAT/URL/网络
3. **P0 拦截**（仅自动）：优先级为 P0 → 终止，提示改用手动模式
4. **Git 检查**：**[🤖]** 工作区不净 → 自动 stash；**[👤]** 工作区不净 → 提示用户处理

### 输出字段

Jira ID、Issue 标题、优先级、执行模式、mcp 状态、Git 状态。进入阶段1。

---

## 阶段1：创建 Git 修复分支

### 命名

`fix/jira-fix-[JIRA-ID]`，如 `fix/jira-fix-YNOTR-12167`

### [🤖 自动] 多项目

**项目识别**（优先级）：触发词显式指定 → 扫描 workspace 一/二级含 `.git` 目录（排除 node_modules/.cursor/dist/build）

**执行**：stash 未提交 → 识别基础分支（优先 release/x.x.x）→ 创建/切换分支。无法识别基础分支时用当前分支并记警告；不在 Git 仓库则报错终止。

### [👤 手动] 单项目

从 URL 提取 ID（正则 `/browse/([A-Z]+-\d+)`）→ 检查分支是否已存在 → 读 `main_branch` 文件创建分支。工作区不净/分支已存在/基础分支不明时中断等用户处理。

### 输出字段

Jira ID、基础分支、修复分支、状态；多项目时表格列出各项目。保存到 `.jira-fix/{JIRA-ID}/00-branch.md`

---

## 阶段2：读取 Jira 信息

### 原则

- 优先获取最新数据（`--live` 模式）
- API 不可用时读取本地缓存
- 复用 `jira-read` skill 的统一逻辑
- **工具限制**：✅ mcp-atlassian API / jira-read；❌ Edit/Write/Bash

调用 `jira-read {JIRA-ID} --live`，降级读本地缓存，再降级手动提供/报错终止。保存到 `.jira-fix/{JIRA-ID}/01-jira-info.md`，更新 state.json。

**提取**：Jira ID、标题、优先级、状态、描述、复现步骤、期望/实际结果、附件、评论。

### 输出字段

Jira ID、标题、优先级、状态、数据来源、问题描述、复现步骤、期望/实际结果、评论摘要。

---

## 阶段3：分析问题

> 只读分析，不修改代码

### 原则

- 深度分析，追问「为什么」至少 3 次
- 结构化输出，中文描述
- **[👤 手动]** 信息不足时主动提问，格式见「通用原则 → 主动提问」
- **工具限制**：✅ Read/Grep/SemanticSearch；❌ Edit/Write/Bash

### 分析维度

- **问题现象**：复现步骤、期望 vs 实际结果
- **代码定位**：Grep/Glob/SemanticSearch 组合搜索关键代码
- **根因分析**：数据流（操作→组件→函数→状态→渲染）+ 调用链，区分直接原因与根本原因
- **影响范围**：功能、平台（Web/Mobile）、模块、连带影响
- **难度预判**：预估改动文件数 + 根因清晰度（清晰/基本清晰/模糊/未知，供阶段3.5使用）

### 输出字段

问题现象（复现/期望/实际）、代码定位（路径:行号+说明）、根因（直接+根本+调用链）、影响范围（模块/平台/连带）、难度预判。保存到 `.jira-fix/{JIRA-ID}/02-analysis.md`

### Red Flags — 禁止行为

- 以「Jira 描述已经很清楚」为由跳过代码定位，直接进入方案评估
- 以「这是常见 bug」为由跳过根因分析，直接复用历史方案
- 分析阶段修改代码（本阶段严格只读，违反阶段工具约束）
- **[🤖 自动]** 所有方案风险 > 中时不暂停，直接执行

---

## 阶段3.5：难度分级 + 模式决策网关

> 基于阶段3的分析结果，使用规则制判定难度等级，决定后续行为

### 🔴 极难触发条件（满足任一即为极难）

| 条件 | 判断依据 |
|------|---------|
| 根因未知 | 分析后仍无法定位根本原因（知道大概模块但无法定位到具体逻辑） |
| 涉及架构变更 | 需要重构模块间通信、变更设计模式、调整核心数据结构 |
| 涉及数据迁移 | 需要数据库 schema 变更或数据格式迁移 |
| API 协议变更 | 需要修改对外接口签名，影响调用方 |
| 改动范围过大 | 预估改动文件 >10 个 或 预估行数变更 >500 行 |
| 跨仓库/跨服务 | 需要在多个独立 Git 仓库中协同修改 |

### 等级判定（未命中极难条件时）

- 预估文件 ≤ 3 个，根因**清晰** → 🟢 **容易**
- 预估文件 4–10 个，根因**基本清晰** → 🟡 **中等**
- 预估文件 ≤ 10 个，根因**较清晰**但改动较多 → 🟠 **困难**

### 模式 × 等级 → 行为

| 等级 | 🤖 自动模式 | 👤 手动模式 |
|------|------------|------------|
| 🟢 容易 | ✅ 正常执行 | 💡 提示「此问题较简单，可切换自动模式」，继续手动流程 |
| 🟡 中等 | ✅ 正常执行 | 💡 提示「此问题风险可控，可切换自动模式」，继续手动流程 |
| 🟠 困难 | ✅ 执行，阶段6完成后暂停等用户审查代码再提交 | ✅ 正常手动流程 |
| 🔴 极难 | ❌ **终止**，输出标记报告 | ⚠️ **风险提示**，用户选择 A/B |

### 输出字段

难度等级（🟢/🟡/🟠/🔴）、命中条件列表、行为提示（终止报告/风险选项 A-B/建议切换模式）、后续行动建议。分级结果保存到 `.jira-fix/{JIRA-ID}/02-grade.md`，更新 state.json `"grade"` 字段

---

## 阶段4：探索与审查方案

### [🤖 自动模式] 自动选择

自动生成 2-3 个方案，优先选：更彻底解决 > 符合规范最佳实践 > 改善代码质量 > 改动较少。

### [👤 手动模式] 等用户选择

若缺少选型偏好，先提出 1～2 个关键问题，再输出方案对比与推荐。

**每个方案必须包含**：核心思路（1-2句）、修改文件/模块、优点（≥2）+ 缺点（≥1）、复杂度+风险等级（低/中/高）、适用场景。

### 输出字段

方案对比表（核心思路/复杂度/风险/推荐度）+ 各方案详情 + 选定/推荐方案及理由。**[🤖 自动]** 自动选定后进入审查；**[👤 手动]** 等用户选定后进入审查。保存到 `.jira-fix/{JIRA-ID}/03-options.md`

### 方案审查（选定方案后执行）

> 对选定方案深度核查；工具限制：✅ Read；❌ Edit/Write/Bash

- **核心逻辑**：实现路径、关键改动点、涉及文件/模块
- **风险清单**：技术风险、兼容性风险、维护成本（各注明影响与缓解建议）
- **迭代**：说「修改/完善/优化方案」可基于审查结果多轮调整

| 场景 | 行为 |
|------|------|
| **[🤖 自动]** 审查风险 ≤ 中 | 输出审查报告，自动进阶段5 |
| **[🤖 自动]** 审查风险 > 中 | 暂停，等用户确认风险可接受 |
| **[👤 手动]** | 输出审查报告，等用户确认后进阶段5 |

### Red Flags — 禁止行为

- 只生成 1 个方案，以「方向已明确」为由跳过方案对比
- **[👤 手动]** 用户未选方案时自行推进到审查
- 跳过方案审查直接进入阶段5（风险未识别）
- 在探索与审查方案阶段修改代码（违反只读约束）

---

## 阶段5：制定计划

> 详细可执行的修改计划

**计划必须包含**：① 根因回顾 ② 方案回顾 ③ 架构设计（可选 Mermaid）④ 文件修改清单（表格：文件/内容/类型）⑤ 修改顺序（考虑依赖关系）⑥ 测试场景（Web/Mobile/兼容性）⑦ 影响范围（服务层/Web/Mobile/风险等级）⑧ 回滚方案

### 确认方式

| 场景 | 行为 |
|------|------|
| **[🤖 自动]** 普通情况 | 自动确认，立即进入阶段6 |
| **[🤖 自动]** 难度🟠困难 或 方案风险>中 | 输出计划后暂停，等用户确认后继续 |
| **[👤 手动]** 普通情况 | 等用户确认 |
| **[👤 手动]** 极难选B | 强制要求用户二次确认「我已知晓风险，继续执行」后才进阶段6 |

保存到 `.jira-fix/{JIRA-ID}/04-plan.md`

### Red Flags — 禁止行为

- 制定计划阶段修改代码（违反只读约束，见「阶段工具约束」表）
- **[🤖 自动]** 困难等级或高风险方案未暂停，直接跳入阶段6
- 计划缺少测试场景或回滚方案就推进到阶段6
- **[👤 手动]** 未等用户确认计划就进入执行

---

## 阶段6：执行计划

> 严格按计划执行，逐步更新 TODO 状态

**前置**：确认在 `fix/jira-fix-[JIRA-ID]` 分支，工作区干净。用 `TodoWrite` 追踪任务状态（pending/in_progress/completed）。

**代码质量**：函数职责单一、命名清晰（函数用动词/变量用名词）、向后兼容。

**修复注释标注**：每处代码改动**必须**在修改行附近添加追踪注释，格式：

```
// fix [JIRA-ID]
```

示例：`// fix YNOTR-12721`

- 注释紧邻修改的代码块（修改前一行或同行尾部均可）
- 多处改动各自标注，不可只在文件顶部统一写一次
- 语言注释符随语言变化（Python 用 `#`，HTML 用 `<!-- -->`，SQL 用 `--`）

### 代码质量检查

1. **Linter 检查**：修改后用 `ReadLints` 检查，新引入的错误必须修复
2. **TypeScript 检查**（自动检测）：项目存在 `tsconfig.json` 时自动调用 `typescript-check` skill，新引入类型错误必须修复

### [🤖 自动模式] 额外能力

**多项目代码修改**：根据文件路径前缀定位对应项目，各项目独立检查 Linter，记录每个项目修改文件及行数统计。

**报告生成**：执行完成后自动生成 `reports/[JIRA-ID]-analysis.md`，包含：基本信息、问题分析、根因、解决方案、修改文件清单、遇到的问题、测试建议。

### 执行后行为

| 场景 | 行为 |
|------|------|
| **[🤖 自动]** 普通情况 | 自动进入阶段7 |
| **[🤖 自动]** 难度🟠困难 | 暂停，输出执行摘要，等用户审查代码后确认继续 |
| **[👤 手动]** 普通情况 | 等用户确认后进入阶段7 |
| **[👤 手动]** 极难选B | 暂停，等用户审查代码，不自动提交 |

### 执行报告

执行完成后必须输出：已修改文件及关键改动说明、偏离计划的调整（如有需说明原因）、验证要点/自查清单。保存到 `.jira-fix/{JIRA-ID}/05-execution.md`

### Red Flags — 禁止行为

- 跳过 Linter 检查，以「只是逻辑改动」为由认为不必要
- 代码改动未添加 `// fix [JIRA-ID]` 注释，导致修复点无法追踪
- 偏离计划修改了计划外文件，未说明原因
- **[🤖 自动]** 遇到计划外问题未暂停，自行决策继续
- 执行完成未输出执行报告直接进入阶段7

---

## 阶段7：提交与回顾

### [🤖 自动模式] 自动提交

1. 收集提交信息（Jira ID、根因、修复方案、修改文件、报告路径）
2. 调用 `git-commit` SKILL（execute=true，自动模式）：自动 add / commit / push
3. **Jira 状态自动回写**：调用 `jira_get_transitions`，匹配「修复完成」（即流转到"已修复"状态）；**禁止流转到"关闭"或"验证通过"**——研发角色只能将 issue 标记为"已修复"，"关闭"由 QA 验证后操作；无匹配时跳过记警告；调用 `jira_add_comment` 写修复评论

**修复评论字段**：修复分支、Commit、根因、修复方案、修改文件、分析报告路径。

### [👤 手动模式] 生成命令

收集提交信息，调用 `git-commit` SKILL 生成命令（不自动执行），生成验证场景（功能/边界/回归各 ≥2），等用户确认后执行。修复评论同自动模式，额外包含验证场景章节。

### Commit Message 格式

```
<type>(<scope>): <subject> <Jira-ID>
```

示例：`fix(ai-summary): 修复分享链接中AI摘要按钮显示问题 YNOTR-12167`

**Type**：fix、feat、refactor、perf、style、docs、test　**Scope 示例**：ai-summary、share、auth、api、ui、core

### 完成输出字段

修复分支、Commit Hash、推送状态、Jira 回写状态、分析报告路径；多项目时用表格。**错误处理**：回写失败不阻断流程，仅输出警告并记录到报告。

保存到 `.jira-fix/{JIRA-ID}/06-report.md`

### Red Flags — 禁止行为

- **[👤 手动]** 未等用户确认直接执行 git push
- Jira 回写失败后未输出警告，静默跳过
- Commit message 遗漏 Jira ID
- 多项目场景只提交了部分项目

---

## 安全机制（自动模式）

修改前自动 stash；**警告阈值**：>10文件 或 >500行；**阻断阈值**：>20文件 或 >1000行（需 `--force` 解除）；Linter 错误自动阻断；记录所有自动决策日志。

---

## 常见错误

| 错误 | 后果 | 修正 |
|------|------|------|
| 跳过代码定位直接给方案 | 方案治标不治本 | 返回阶段3完成根因定位 |
| 自动模式遇极难仍继续执行 | 高风险改动失控 | 触发阶段3.5网关，终止流程 |
| 手动模式跳过方案确认直接制定计划 | 执行错误方案 | 等用户选择后再进阶段5 |
| 执行完成未做 Linter 检查 | 引入新的代码错误 | ReadLints 是必须步骤，不可跳过 |
| 代码改动缺少 `// fix [JIRA-ID]` 注释 | 修复点无法追踪，Code Review 难以定位 | 每处改动附近必须标注 |
| 分析阶段使用 Edit/Write 修改代码 | 破坏只读约束 | 见「阶段工具约束」表 |

## 批量修复

将每个 Jira bug 作为独立任务，交由**任务编排工具**（如 `/ralph`、`/autopilot`、`/ultrawork`，以当前平台为准）依次调用 `jira-fix [URL]`。编排工具负责：单个失败不阻断后续、跟踪整体进度、汇总结果。

**示例（以 /ralph 为例）**：

```
/ralph 批量修复以下 Jira bug：
YNOTR-12167 YNOTR-12168 https://your-jira.example.com/browse/YNOTR-12169
```
