# Jira Bug 修复工作流

> 默认自动模式，全流程无需确认。说「手动模式」或「手动修复」进入手动确认模式。

## 触发词与模式

| 说法示例 | 模式 | 说明 |
|---------|------|------|
| 「修复这个 bug [URL]」「帮我修复 [URL]」`jira-fix [URL]` | 🤖 自动 | 默认，全流程自动执行 |
| 「手动修复 [URL]」「修复 [URL] 手动模式」`jira-fix [URL] --manual` | 👤 手动 | 方案/计划/提交需用户确认 |
| 「强制修复 [URL]」「跳过分级修复 [URL]」`jira-fix [URL] --force` | 🤖 自动 | 跳过难度分级，强制执行 |
| 「继续修复」「从上次继续」「恢复修复 [URL]」`jira-fix [URL] --resume` | 当前模式 | 从断点恢复 |

**模式识别规则**：触发词中含「手动」「我来确认」「一步一步」→ 手动模式；含「强制」「跳过分级」→ 跳过分级；含「继续」「恢复」「上次」→ 断点恢复；其余默认自动模式。

## 模式差异速查

| 阶段 | 🤖 自动模式 | 👤 手动模式 |
|------|------------|------------|
| 阶段0：前置检查 | P0 自动拦截，工作区自动 stash | P0 提示，工作区提示用户处理 |
| 阶段1：创建分支 | 多项目支持，自动 stash，自动切换已有分支 | 单项目，错误等待用户处理 |
| 阶段3.5：难度分级 | 极难 → 终止，输出标记报告 | 极难 → 列出风险，用户选择 A/B |
| 阶段4：评估方案 | AI 自动选择最佳方案 | 输出方案表，等用户选择 |
| 阶段5：制定计划 | 自动确认，直接进阶段6 | 等用户确认 |
| 阶段6：执行计划 | 执行 + Linter + 自动生成分析报告 | 执行 + Linter |
| 阶段7：Git 提交 | 自动 push + Jira 状态回写 | 生成命令 + 验证场景，等用户执行 |
| 中断恢复 | 自动恢复，不询问 | 询问是否恢复 |

## 状态持久化（中断恢复）

### 恢复触发
- 说「继续修复」「从上次继续」「恢复修复 [URL]」，或使用 `jira-fix [URL] --resume`
- 启动时自动检测已有进度：**[🤖 自动]** 自动恢复；**[👤 手动]** 提示是否恢复

### 状态目录
```
.jira-fix/{JIRA-ID}/
├── state.json           ← 进度跟踪（含 mode 字段）
├── 01-jira-info.md      ← 阶段2输出
├── 02-analysis.md       ← 阶段3输出
├── 02-grade.md          ← 阶段3.5难度分级结果
├── 03-options.md        ← 阶段4输出
├── 04-plan.md           ← 阶段5输出
├── 05-execution.md      ← 阶段6输出
└── 06-report.md         ← 阶段7输出
```

### state.json 结构
```json
{
  "jira_id": "YNOTR-12345",
  "jira_url": "https://your-jira.example.com/browse/YNOTR-12345",
  "mode": "auto",
  "current_phase": 3,
  "completed_phases": [0, 1, 2],
  "branch": "fix/jira-fix-YNOTR-12345",
  "grade": null,
  "selected_option": null,
  "started_at": "ISO_TIMESTAMP",
  "last_updated": "ISO_TIMESTAMP"
}
```

### 恢复逻辑
1. 检测到已有 state.json 时：
   - **[🤖 自动]** 自动读取，从 current_phase 继续，不询问
   - **[👤 手动]** 提示「发现未完成任务（阶段X），是否恢复？」
2. 恢复后从已保存的阶段输出文件继续，减少上下文依赖

### 清理
修复完成后保留 `.jira-fix/{JIRA-ID}/` 目录，state.json 设 `current_phase: "completed"`

## 阶段0：前置检查

> 任一检查失败则中断流程

### 执行步骤

1. **确定执行模式**：
   - 解析触发词，识别 `--manual` / `--force` / `--resume` 参数
   - 将 mode 写入 state.json：`"mode": "auto"` 或 `"mode": "manual"`

2. **mcp-atlassian 连通性检查**：
   调用 `jira_get_issue`(issue_key="{JIRA-ID}", fields="summary,priority") 仅获取标题和优先级
   - ✅ 成功：记录 Issue 标题和优先级，继续流程
   - ❌ 失败：**立即终止**，输出：
   ```
   ❌ mcp-atlassian 连接失败

   请检查：
   1. opencode.json 中是否配置了 mcp-atlassian
   2. JIRA_PERSONAL_TOKEN 是否有效
   3. JIRA_URL 是否正确（当前：https://your-jira.example.com）
   4. 网络是否可达
   ```

3. **P0 快速拦截**（仅自动模式）：
   - **[🤖 自动]** 若优先级为 P0 → 立即终止，提示：「P0 级别 bug 需要人工审核，请使用 --manual 模式」

4. **Git 环境检查**：
   - 是否在 Git 仓库中（不在则跳过阶段1）
   - **[🤖 自动]** 工作区不干净 → 自动 `git stash save "Auto-stash before jira-fix"`
   - **[👤 手动]** 工作区不干净 → 提示用户提交/暂存/放弃，等待确认

### 输出格式

```
## 阶段0完成：前置检查通过

**Jira ID**: YNOTR-12167
**Issue 标题**: [标题]（已确认存在）
**优先级**: P1
**执行模式**: 🤖 自动 / 👤 手动
**mcp-atlassian**: ✅ 已连接
**Git 仓库**: ✅

---
进入阶段1：创建 Git 修复分支
```

## 阶段1：创建 Git 修复分支

### 原则
- 从 `main_branch` 文件读取主开发分支，不存在则使用当前分支
- 统一命名：`fix/jira-fix-[JIRA-ID]`

### [🤖 自动模式] 多项目支持

**项目识别策略**（按优先级）：
1. 触发词中显式指定：`jira-fix [URL] project1,project2`
2. 自动扫描 workspace 根目录一/二级子目录（存在 `.git`）
3. 排除：`node_modules`、`.cursor`、`dist`、`build`、`.git`

**自动执行步骤**：
1. 检查 Git 状态，自动 stash 未提交修改
2. 自动识别基础分支（优先 release/x.x.x）
3. 自动提取 Jira ID 并创建分支
4. 分支已存在时自动切换

**自动错误处理**：
- 未提交修改：自动 `git stash save "Auto-stash before jira-fix"`
- 分支已存在：自动 `git checkout fix/jira-fix-[JIRA-ID]`
- 无法识别基础分支：使用当前分支，记录警告
- 不在 Git 仓库：报错终止

### [👤 手动模式] 单项目

**执行步骤**：
1. 检查 Git 状态和当前分支
2. 从 Jira URL 提取 ID（正则：`/browse/([A-Z]+-\d+)`）
3. 检查修复分支是否已存在，存在则中断提示用户
4. 读取 `main_branch` 文件（不存在则使用当前分支），创建修复分支

**错误处理**：
- 未提交修改：提示用户提交/暂存/放弃，等待确认
- 分支已存在：中断，提示用户手动处理
- 无法识别基础分支：列出所有分支，请用户指定

### 命名规范

| Jira URL 示例 | 分支名称 |
|--------------|---------|
| `.../browse/YNOTR-12167` | `fix/jira-fix-YNOTR-12167` |
| `.../browse/PROJ-123` | `fix/jira-fix-PROJ-123` |

前缀 `fix/jira-fix-` + Jira ID（原样保留大小写）

### 输出格式

**[🤖 自动] 单项目**：
```
## 阶段1完成：Git分支已自动创建

**Jira ID**: YNOTR-12167
**基础分支**: release/8.2.30
**修复分支**: fix/jira-fix-YNOTR-12167
**状态**: 已自动切换到修复分支

---
自动进入阶段2：读取Jira信息
```

**[🤖 自动] 多项目**：
```
## 阶段1完成：Git分支已自动创建（多项目）

| 项目 | 基础分支 | 修复分支 | 状态 |
|------|---------|---------|------|
| backend | release/8.2.30 | fix/jira-fix-YNOTR-12167 | ✅ 已创建 |
| frontend | release/8.2.30 | fix/jira-fix-YNOTR-12167 | ✅ 已创建 |

---
自动进入阶段2：读取Jira信息
```

**[👤 手动]**：
```
## 阶段1完成：Git分支已创建

**Jira ID**: YNOTR-12167
**基础分支**: release/8.2.30（来源：main_branch 文件）
**修复分支**: fix/jira-fix-YNOTR-12167

---
准备进入阶段2：读取 Jira 信息
```

输出保存到 `.jira-fix/{JIRA-ID}/00-branch.md`

## 阶段2：读取 Jira 信息

### 原则
- 优先获取最新数据（`--live` 模式）
- API 不可用时读取本地缓存
- 复用 `jira-read` skill 的统一逻辑

### 执行步骤
1. 调用 `jira-read {JIRA-ID} --live`（直接从 API 获取最新数据）
2. jira-read 自动处理：
   - live 模式：调用 mcp-atlassian API 获取最新数据并更新缓存
   - 降级：API 不可用时读取本地缓存
   - 再降级：**[👤 手动]** 请用户手动提供；**[🤖 自动]** 报错终止
3. 提取结构化信息
4. 保存到 `.jira-fix/{JIRA-ID}/01-jira-info.md`，更新 state.json

### 提取信息清单

**必需**：Jira ID、标题、优先级（P0-P4）、状态

**详细**：问题描述、复现步骤、期望结果、实际结果、附件链接、评论历史

### 输出格式

```
## 阶段2完成：Jira信息已读取

**Jira ID**: YNOTR-12167
**标题**: [标题]
**优先级**: P1
**状态**: 待处理
**数据来源**: API 最新 / 本地缓存

**问题描述**: [描述]
**复现步骤**: 1. ... 2. ...
**期望结果**: [期望]
**实际结果**: [实际]

**评论摘要**: 共 X 条
> 最新评论 - [用户]（[时间]）: [内容摘要]

---
[🤖 自动] 自动进入阶段3：分析问题
[👤 手动] 准备进入阶段3：分析问题
```

## 阶段3：分析问题（只读）

> 只读分析，不修改代码

### 原则
- 深度分析，追问「为什么」至少 3 次
- 结构化输出，中文描述
- **[👤 手动]** 信息不足时主动提出 1～3 个关键问题，采用「提问与简答约定」（用户可回复如 1A、2B）

### 分析维度

**1. 问题现象**：Jira 信息、复现步骤、期望 vs 实际结果

**2. 代码定位**：使用 Grep/Glob/SemanticSearch 组合搜索关键代码

**3. 根因分析**：
- 数据流分析：用户操作→组件→函数→状态→渲染
- 调用链分析：追踪完整函数调用链
- 区分直接原因和根本原因

**4. 影响范围**：功能影响、平台影响（Web/Mobile）、模块影响、连带影响

**5. 难度预判**：
- 预估改动文件数
- 根因清晰度：清晰 / 基本清晰 / 模糊 / 未知
（供阶段3.5使用）

### 输出格式

```
## 阶段3：分析完成

### 1. 问题现象
**复现步骤**: 1. ... 2. ...
**期望结果**: [期望]
**实际结果**: [实际]

### 2. 相关代码定位
- `path/file.js:行号` - [说明]

### 3. 根因分析
**直接原因**: [描述]
**根本原因**: [描述]
**调用链**: [流程]

### 4. 影响范围
**受影响模块**: [列表]
**平台**: Web端 / Mobile端
**连带影响**: [列表]

### 5. 难度预判
**预估改动文件数**: X 个
**根因清晰度**: 清晰 / 基本清晰 / 模糊 / 未知

---
进入阶段3.5：难度分级
```

保存到 `.jira-fix/{JIRA-ID}/02-analysis.md`

## 阶段3.5：难度分级 + 模式决策网关

> 基于阶段3的分析结果，使用规则制判定难度等级，决定后续行为

### 🔴 极难触发条件（满足任一即为极难）

| 条件 | 判断依据 |
|------|---------|
| 根因未知 | 分析后仍无法定位根本原因（知道大概模块但无法定位到具体逻辑） |
| 涉及架构变更 | 需要重构模块间通信、变更设计模式、调整核心数据结构 |
| 涉及数据迁移 | 需要数据库 schema 变更或数据格式迁移 |
| API 协议变更 | 需要修改对外接口签名，影响调用方 |
| 改动范围过大 | 预估改动文件 >10 个 或 预估行数变更 >500 行 |
| 跨仓库/跨服务 | 需要在多个独立 Git 仓库中协同修改 |

### 等级判定（未命中极难条件时）

- 预估文件 ≤ 3 个，根因**清晰** → 🟢 **容易**
- 预估文件 4–10 个，根因**基本清晰** → 🟡 **中等**
- 预估文件 ≤ 10 个，根因**较清晰**但改动较多 → 🟠 **困难**

### 模式 × 等级 → 行为

| 等级 | 🤖 自动模式 | 👤 手动模式 |
|------|------------|------------|
| 🟢 容易 | ✅ 正常执行 | 💡 提示「此问题较简单，可切换自动模式」，继续手动流程 |
| 🟡 中等 | ✅ 正常执行 | 💡 提示「此问题风险可控，可切换自动模式」，继续手动流程 |
| 🟠 困难 | ✅ 执行，阶段6完成后暂停等用户审查代码再提交 | ✅ 正常手动流程 |
| 🔴 极难 | ❌ **终止**，输出标记报告 | ⚠️ **风险提示**，用户选择 A/B |

### [🤖 自动模式] 极难 → 终止报告

```
## ⚠️ jira-fix 已终止：难度超出自动修复阈值

**Jira ID**: YNOTR-12167
**难度等级**: 🔴 极难
**命中条件**:
  - [x] 根因未知：分析后无法定位具体修改点
  - [ ] 涉及架构变更
  - [ ] 改动范围过大

**已完成工作**:
  - 分支已创建：fix/jira-fix-YNOTR-12167
  - 分析已保存：.jira-fix/YNOTR-12167/02-analysis.md

**建议后续行动**:
  1. 使用手动模式继续：jira-fix [URL] --manual --resume
  2. 跳过分级强制执行：jira-fix [URL] --force（风险自担）
  3. 团队评审后再修复
```

### [👤 手动模式] 极难 → 风险提示 + 选项

```
## ⚠️ 高风险提示：当前问题难度评级为「极难」

**命中条件**:
  - [x] [触发条件名称]: [具体说明]

**建议**：

  选项 A — 仅保留分析（推荐）
    保存分析至 .jira-fix/YNOTR-12167/02-analysis.md
    在 Jira 添加评论「AI 分析完成，需人工评估后继续修复」
    结束本次修复，等待人工接手

  选项 B — 继续执行（风险自担）
    阶段5制定计划后，强制要求二次确认才可进入阶段6
    阶段6执行完成后不自动提交，等用户审查代码后手动提交

请回复 A 或 B：
```

### [👤 手动模式] 容易/中等 → 建议切换自动

```
💡 提示：此问题难度评级为「容易/中等」
预估改动 [X] 个文件，根因明确，风险可控。
可切换自动模式：「修复这个 bug [URL]」（不加手动模式即可）

继续手动流程请直接输入「继续」。
```

分级结果保存到 `.jira-fix/{JIRA-ID}/02-grade.md`，更新 state.json `"grade"` 字段

## 阶段4：评估方案

### [🤖 自动模式] 自动选择

自动生成 2-3 个方案，按以下优先级选择：
1. 更彻底解决问题（而非临时修复）
2. 符合代码规范和最佳实践
3. 改善代码质量（重构优于补丁）
4. 复杂度相近时选改动较少的

```
if (所有方案风险 > 中) {
  → 继续执行，但阶段5后强制等用户确认
} else {
  → 选择推荐度最高的方案
  → 推荐度相同时选复杂度最低的
}
```

### [👤 手动模式] 等用户选择

若缺少选型偏好，先提出 1～2 个关键问题，再输出方案对比与推荐。

### 每个方案必须包含

1. 核心思路（1-2 句话）
2. 需要修改的文件/模块
3. 优点（≥2）、缺点（≥1）
4. 复杂度（低/中/高）、风险等级（低/中/高）
5. 适用场景

### 输出格式

**[🤖 自动]**：
```
## 阶段4：方案已自动选择

| 方案 | 核心思路 | 复杂度 | 风险 | 推荐度 |
|------|---------|--------|------|--------|
| 方案1 | [思路] | 低 | 低 | ⭐⭐⭐⭐⭐ |
| 方案2 | [思路] | 中 | 低 | ⭐⭐⭐ |

**AI 自动选择**：方案1
**选择理由**: 1. ... 2. ...

---
自动进入阶段5：制定计划
```

**[👤 手动]**：
```
## 阶段4：评估方案

[方案对比表]
[每个方案详细说明]

### 推荐方案：方案1
理由：...

---
请选择方案编号，进入阶段5：制定计划
```

保存到 `.jira-fix/{JIRA-ID}/03-options.md`

## 阶段5：制定计划

> 详细可执行的修改计划

### 计划必须包含 8 个部分

1. 问题根因回顾（阶段3）
2. 选定方案回顾（阶段4）
3. 架构设计（可选 Mermaid 图）
4. 文件修改清单（表格：文件、内容、类型）
5. 修改顺序（考虑依赖关系）
6. 测试场景（Web端/Mobile端/兼容性）
7. 影响范围（服务层/Web/Mobile/风险等级）
8. 回滚方案

### 确认方式

## 阶段7：Git 提交代码

### [🤖 自动模式] 自动提交

**执行步骤**：
1. 收集提交信息（Jira ID、根因、修复方案、修改文件、报告路径）
2. 调用 `git-commit` SKILL（execute=true，自动模式）：自动 add / commit / push
3. **Jira 状态自动回写**：
   - 调用 `jira_get_transitions`(issue_key="{JIRA-ID}") 获取可用状态
   - 执行状态转换：优先匹配 "Code Review" > "In Review" > "In Progress"
   - 无匹配时跳过，记录警告
   - 调用 `jira_add_comment`(issue_key="{JIRA-ID}", body=修复评论)

## 核心原则

### 问题定位
- 深度优先：追问「为什么」至少 3 次，找到根本原因
- 全面分析：完整数据流、调用链、影响范围
- 证据驱动：基于代码和测试，不凭猜测

### 方案评估
- 多样化：至少 2 个方案，包含不同类型
- 客观性：优缺点都说明，不隐藏风险
- 权衡考虑：复杂度 vs 效果、短期 vs 长期

### 代码修改
- 职责单一：一个函数一个职责
- 命名清晰：见名知意，动词 + 名词
- 概念分离：区分相似概念
- 向后兼容：保留旧接口，考虑旧数据

### 质量保证
- Linter 零容忍：新引入的错误必须修复
- 多端一致：Web 和 Mobile 同步修改
- 提交规范：生成完整 commit message

## 安全机制（自动模式）

1. 修改前自动 stash：`git stash save "Auto-stash before jira-fix"`
2. **改动阈值检测**：
   - 警告：修改 >10 个文件 或 >500 行
   - 阻断：修改 >20 个文件 或 >1000 行（需 `--force` 解除）
3. Linter 错误自动阻断流程
4. 详细操作日志，记录所有自动决策

## 批量修复

批量修复需要借助**任务编排工具**——即能将一组任务拆解为独立子任务、逐个执行并跟踪状态的工具。不同平台的命令名称可能不同（如 `/ralph`、`/autopilot`、`/ultrawork` 等），使用时以当前平台支持的编排工具为准。

### 核心思路

将每个 Jira bug 描述为一个独立任务，交由编排工具依次调用 `jira-fix [URL]` 完成修复。编排工具负责：
- 逐个执行，单个失败不阻断后续
- 跟踪整体进度，支持断点恢复
- 完成后汇总结果

### 示例（以 /ralph 为例）

```
/ralph 批量修复以下 Jira bug：
YNOTR-12167
YNOTR-12168
https://your-jira.example.com/browse/YNOTR-12169
```
