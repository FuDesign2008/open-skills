# Jira 方案探索与计划制定（阶段3～4）

> 本 skill 由 `jira-fix-workflow` 在**阶段3和阶段4**调用，请勿独立使用。
> **工具限制**：✅ Read（查看代码细节）；❌ Edit/Write/Bash

---

## 阶段3：探索与审查方案

**🔌 增强能力集成**：若环境探索发现了「💡 方案设计」类能力（如 `brainstorming`、OMC `analyst`/`architect` agent 等），在方案探索环节加载并辅助多方案对比分析。

### 方案选择

**[🤖 自动]** 自动生成 2-3 个方案，优先选：更彻底解决 > 符合规范最佳实践 > 改善代码质量 > 改动较少。

**[👤 手动]** 若缺少选型偏好，先提出 1 个最关键的问题，得到回答后再输出方案对比与推荐。

### 输出格式

1. **开头**：方案对比表
2. **中间**：每个方案详细展开（见下方「每个方案包含」）
3. **结尾**：再次输出方案对比表，便于快速回顾

| 方案 | 描述 | 优点 | 缺点 | 复杂度 | 推荐度 |
|------|------|------|------|--------|--------|
| 方案1 | ... | ... | ... | 低/中/高 | ⭐⭐⭐⭐⭐ |
| 方案2 | ... | ... | ... | 低/中/高 | ⭐⭐⭐ |

### 每个方案包含

- 核心思路（1-2 句话）
- 需要修改的文件/模块
- 优点（≥2）、缺点（≥1）
- 复杂度 + 风险等级（低/中/高）、适用场景

**[🤖 自动]** 自动选定最优方案后进入审查；**[👤 手动]** 等用户选定后进入审查。保存到 `.jira-fix/{JIRA-ID}/03-options.md`

> ⛔ **[手动模式 — 阶段3 方案选择出口]** 输出方案对比表后**立即停止**，在表末**必须附上**：
> 「请选择一个方案（回复方案编号，如「方案1」），AI 将进入方案审查。」
> 🤖 **[自动模式]** 自动选定最优方案，直接进入审查。

---

### 方案审查 — 循环审查机制

> 对选定方案深度核查，循环审查直到通过；工具限制：✅ Read；❌ Edit/Write/Bash

#### 审查维度（每轮必须覆盖）

1. **解决有效性**：是否完整覆盖根因、能否高效高质量解决问题、核心逻辑与关键实现点
2. **副作用与风险**：功能副作用 + 非功能副作用（性能/安全/可维护性）、已识别问题有无缓解措施
3. **实现可行性**：改动范围、依赖关系、涉及文件/模块是否明确可执行
4. **代码规范符合度**：是否符合项目现有模式和最佳实践

#### 审查结论（二级制）

| 结论 | 判定标准 | 后续动作 |
|------|---------|---------|
| ✅ **通过** | 四项审查维度均无阻断问题，仅存在可接受的低风险 | 进入阶段4 |
| ❌ **不通过** | 任一维度存在需解决的问题或不可接受的风险 | 进入优化→重新审查循环 |

**阻断问题判定指引**（满足任一即为 ❌ 不通过）：

- 方案无法完整覆盖阶段2确定的根因（核心逻辑不完整）
- 存在已识别但未提出缓解措施的中/高风险
- 涉及的修改文件或依赖关系不明确，无法据此制定可执行计划
- 与项目现有设计模式或编码规范明显冲突
- 可能引入新 bug 或破坏现有功能的副作用未被处理

**非阻断问题**（可标注为建议，但不阻止通过）：已有缓解措施的低风险项、代码风格偏好、可后续迭代的性能改进

#### 循环流程

```
方案选定 → 审查（第 N 轮）→ 输出审查报告
                ↓
          审查结论判定
           ├── ✅ 通过 → 进入阶段4
           └── ❌ 不通过 → 方案优化 → 回到审查（第 N+1 轮）
                              ↓
                        达到循环上限？
                         ├── 否 → 继续循环
                         └── 是 → 超限处理
```

#### [🤖 自动模式] 审查循环

- **判定者**：AI 自动判定通过/不通过
- **不通过时**：AI 根据审查报告中的问题清单**自动优化方案**，输出优化说明，然后重新审查
- **循环上限**：**3 轮**（第 1 轮为初次审查，最多优化 2 次后重审）
- **超限处理**：
  - **非批量模式**：暂停，输出 3 轮审查摘要（每轮的问题与优化记录），等用户介入决策（「继续审查」/ 「重选方案」/ 「手动调整方向后重审」）
  - **批量模式**：标记该 issue 为「审查未通过（超限）」，跳过后续阶段，记入批量报告

#### [👤 手动模式] 审查循环

- **判定者**：用户判定通过/不通过
- AI 输出审查报告后，等用户响应：
  - 「通过」「确认」「OK」→ 进入阶段4
  - 「修改方案」「完善方案」「优化方案」→ 按用户指导优化，重新审查
  - 「重选方案」→ 回到方案选择环节
- **循环上限**：无（用户自主控制）

#### 审查报告输出字段（每轮必须输出）

- **审查轮次**：第 N 轮（共 M 轮上限）
- **四维度评估**：各维度标注 ✅ 或 ❌ + 具体问题
- **问题清单**：不通过时，编号 + 问题描述 + 严重程度
- **审查结论**：✅ 通过 / ❌ 不通过
- **[❌ 不通过时]** 优化建议：针对每个问题的具体改进方向

保存到 `.jira-fix/{JIRA-ID}/03-options.md`（追加审查记录）

> ⛔ **[手动模式 — 阶段3 审查出口]** 输出审查报告后**立即停止**，在报告末**必须附上**：
> 「⏸️ 方案审查完成。是否进入**阶段4：制定计划**？回复「确认」继续，或说「修改方案」重新审查。」
> 🤖 **[自动模式]** AI 自动判定通过/不通过；通过进入阶段4，不通过自动优化后重审（上限3轮，超限暂停）。

## 阶段4：制定计划

**🔌 增强能力集成**：若发现「📝 计划制定」类能力（如 `writing-plans`、OMC `planner` agent 等），加载并辅助生成更结构化的执行计划。

**计划必须包含**：① 根因回顾 ② 方案回顾 ③ 架构设计（可选 Mermaid）④ 文件修改清单（表格：文件/内容/类型）⑤ 修改顺序（考虑依赖关系）⑥ 测试场景（Web/Mobile/兼容性）⑦ 影响范围（服务层/Web/Mobile/风险等级）⑧ 回滚方案

### 确认方式

| 场景 | 行为 |
|------|------|
| **[🤖 自动]** 普通情况 | 自动确认，立即进入阶段5 |
| **[🤖 自动]** 难度🟠困难 或 方案风险>中 | 输出计划后暂停，等用户确认 |
| **[👤 手动]** 普通情况 | 等用户确认 |
| **[👤 手动]** 极难选B | 强制要求用户二次确认「我已知晓风险，继续执行」后才进阶段5 |

保存到 `.jira-fix/{JIRA-ID}/04-plan.md`

> ⛔ **[手动模式 — 阶段4 出口]** 输出计划后**立即停止**，在计划末**必须附上**：
> 「⏸️ 阶段4（制定计划）完成。是否进入**阶段5：执行计划**？回复「确认」继续，或说明需要调整。」
> 🤖 **[自动模式]** 普通情况自动确认；困难等级或方案风险>中时**暂停**等用户确认后再进入阶段5。

---

## Red Flags — 禁止行为

- 只生成 1 个方案，以「方向已明确」为由跳过方案对比
- **[👤 手动]** 用户未选方案时自行推进到审查
- 跳过方案审查直接进入阶段4（风险未识别）
- 在本 skill 执行期间修改代码（违反只读约束）
- **[🤖 自动]** 审查不通过时不优化方案，直接进入阶段4
- **[🤖 自动]** 审查循环超过 3 轮上限仍不暂停
- **[🤖 自动]** 优化方案时未输出优化说明，导致审查记录不可追溯
- **[👤 手动]** 用户未明确判定通过/不通过，AI 自行推进
- 计划缺少测试场景或回滚方案就推进到阶段5
- **[🤖 自动]** 困难等级或高风险方案未暂停，直接跳入阶段5
