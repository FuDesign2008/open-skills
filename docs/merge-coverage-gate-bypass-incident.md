# 合并前覆盖率门控被绕过 —— 事件复盘与 Skill 优化建议

> **用途**：本文档记录一次 opsx-solve-workflow「合并前覆盖率门控」被绕过的真实事件，分析根因，供 Skill 作者优化 `opsx-solve-workflow` / `jira-fix-workflow` / `opsx-jira-fix-workflow` / `test-coverage-analyzer` 等相关 Skill 时参考。

> **事件日期**：2026-07-22
> **涉及工程**：markdown-editor（`/Users/admin/workspace/ynote/special-markdown/markdown-editor`）
> **涉及 MR**：[#450 feat(file-link): 新增协同表格 icon 并对齐 bulb 扩展名映射](https://g.hz.netease.com/cowork/web/ynote/markdown-editor/-/merge_requests/450)

## 一、事件经过

### 1.1 背景

通过 `opsx-solve-workflow` 完成「协同表格 icon + 对齐 bulb 扩展名映射 + 补充 spec/单测」任务，流程推进到阶段 8（回顾归档），change 已归档，分支 `fuyg/8.2.80-collab-table-icon` 已 rebase 到最新 `release/8.2.80` 并 force push。

### 1.2 经过

1. 用户说「**准备 merge MR**」。
2. AI 未运行 `test-coverage-analyzer` 门控，直接执行 `glab mr view 450` 确认状态 → `glab mr merge 450 --yes`。
3. MR #450 fast-forward 合并成功（Pipeline succeeded）。
4. 合并后用户问「MR 合并前走了吗？」（指覆盖率门控）。
5. AI 才意识到跳过了门控，如实承认「没有走」，补跑 `test-coverage-analyzer --base a6a6fa8` 留档（变更覆盖率 100%，但已无门控拦截价值）。

### 1.3 后果

- **门控失效**：合并前覆盖率门控本应在合并前运行，拦截不达标代码。MR 已合并，门控失去拦截意义，只能补跑留档。
- **流程违规**：违反 opsx-solve-workflow 阶段 8「合并前覆盖率门控（强制）」规范。
- **无法回退**：MR 已合并到 `release/8.2.80`（保护分支），无法撤销。

## 二、根因分析

### 2.1 当前门控规则的设计前提

`opsx-solve-workflow/SKILL.md` 阶段 8 的门控规则（第 524-532 行）：

> **顺序约束**：归档 + diff 检查 → 分支收尾决策 → **仅当决策为「合并」时**触发合并前覆盖率门控（见下）→ 执行合并。选择「保留当前分支」「继续开发」**不触发**门控。

`reference.md` 第 55 行：

> **触发条件**：用户在分支收尾/合并决策中已选定「合并」。本门控在合并执行前运行，是合并的强制前置。

**设计前提**：门控的触发锚定在「分支收尾决策」这个环节——AI 必须先走 `finishing-a-development-branch` 的分支收尾决策流程，用户在决策中选定「合并」，才触发门控。

### 2.2 实际场景的脱节

真实场景中，用户常常**跳过「分支收尾决策」环节**，直接下达合并指令：

- 「准备 merge MR」
- 「合并 MR」
- 「merge 一下」

此时：

1. **AI 不在 opsx 流程上下文里**：用户说「准备 merge MR」时，AI 的判断焦点是「执行合并动作」，而非「继续 opsx 阶段 8 的分支收尾决策」。
2. **无独立触发规则**：Skill 没有定义「合并动作本身 = 必须先跑门控」的独立规则。门控规则只挂在「分支收尾决策为合并」这个状态上，而该状态在本场景从未被进入。
3. **`glab mr merge` 无前置检查**：AI 直接调用 `glab mr merge`，没有任何机制提醒「合并前需先跑覆盖率门控」。

### 2.3 触发条件的设计缺陷

当前触发条件「**用户在分支收尾/合并决策中已选定「合并」**」存在两个漏洞：

| 漏洞 | 说明 |
|------|------|
| **依赖未发生的状态** | 假设 AI 一定会走「分支收尾决策」流程，但用户可直接下达合并指令绕过该流程 |
| **触发锚点过窄** | 只锚定「分支收尾决策」环节，未覆盖「用户直接说 merge/合并」这一更常见的触发路径 |

## 三、当前规则的其他观察

### 3.1 门控规则本身是完善的

`reference.md` 的门控步骤（构造 `--base`、调用脚本、判定矩阵、显式跳过留痕、模式生命周期）设计完善，判定矩阵覆盖了达标/不达标/崩溃/无测试代码四种情况。**问题不在门控执行逻辑，而在触发判断**。

### 3.2 「显式跳过」留痕机制存在但未被利用

`reference.md` 第 78-79 行定义了显式跳过留痕：

> `【覆盖率门控跳过】用户显式跳过，未运行 test-coverage-analyzer。时间：<ISO 时间戳>。决策人：用户。`

但本次事件不是「用户显式跳过」，而是「AI 未意识到要跑门控」——属于**隐式漏跑**，留痕机制无法捕获。

### 3.3 前置检测的降级过于宽松

`reference.md` 第 57-59 行：

> **前置检测**：环境探索是否发现 `test-coverage-analyzer` skill？
> - ❌ 未发现 → 静默跳过本门控，直接执行合并（不报错、不阻断、不留痕）

「未发现 skill → 静默跳过且不留痕」过于宽松，本次事件若发生在未做环境探索的上下文中，即便有触发也会被静默跳过。

## 四、优化建议（给 Skill 作者）

### 建议 1：把「合并动作」本身作为门控触发锚点（核心建议）

**问题**：当前触发锚点是「分支收尾决策为合并」，过于依赖流程上下文。

**优化**：在 `SKILL.md` 和 `reference.md` 增加独立的「合并动作触发规则」，**不依赖**是否走过分支收尾决策：

> **合并动作触发规则**（独立于分支收尾决策）：当 AI 即将执行任何合并动作（`glab mr merge` / `gh pr merge` / `git merge` 到保护分支等）时，**必须**先运行合并前覆盖率门控。无论该合并指令来自：
> - 分支收尾决策中用户选定「合并」
> - 用户直接下达「merge MR / 合并 MR / 准备合并」指令
> - 其他任何触发合并的场景
>
> 判定「即将执行合并动作」的关键信号：AI 准备调用 `glab mr merge`、`gh pr merge`，或用户消息含「merge」「合并」「准备合并」等词。

这样即使跳过分支收尾决策，合并动作本身也会触发门控。

### 建议 2：在 `glab mr merge` / `gh pr merge` 前置加显式提醒

**问题**：AI 直接调用合并命令时无任何前置检查。

**优化**：在 SKILL.md 的合并相关章节加粗提醒：

> ⚠️ **合并命令前置检查（强制）**：调用 `glab mr merge` / `gh pr merge` 前，AI 必须自问「是否已运行合并前覆盖率门控？」。若未运行，**先跑门控再合并**，不得直接执行合并命令。

### 建议 3：区分「显式跳过」与「隐式漏跑」

**问题**：当前只有「用户显式跳过」的留痕，无法捕获「AI 未意识到要跑」的隐式漏跑。

**优化**：在门控判定矩阵增加一行：

| 执行结果 | 处理 |
|---|---|
| 🕳️ AI 即将合并但未运行门控（隐式漏跑） | **视为门控未通过**，暂停合并，补跑门控；若已合并则如实报告漏跑并留痕 |

### 建议 4：收紧「未发现 skill」的降级

**问题**：「未发现 test-coverage-analyzer → 静默跳过且不留痕」过于宽松。

**优化**：改为「未发现 → 提示用户门控不可用，**留痕**说明未跑门控的原因，再由用户决定是否继续合并」，避免静默跳过。

### 建议 5：门控触发判断的检查清单化

**问题**：触发判断散落在文字描述中，AI 容易遗漏。

**优化**：在 reference.md 提供一个显式的「合并前检查清单」，AI 在执行合并前必须逐项确认：

```
合并前检查清单（AI 执行合并前必须完成）：
- [ ] 当前是否在 opsx/jira-fix 工作流上下文？
  - 是 → 按阶段 8 分支收尾决策流程，决策为「合并」时触发门控
  - 否 → 进入「合并动作触发规则」（见建议 1）
- [ ] 是否已运行 test-coverage-analyzer 门控？
  - 否 → 先跑门控（除非用户显式跳过并留痕）
- [ ] 门控结果是否达标？
  - 达标 → 继续合并
  - 不达标/崩溃/无报告 → 暂停，等用户决策
- [ ] 若用户显式跳过 → 是否已写入留痕？
```

### 建议 6：跨 Skill 触发（test-coverage-analyzer 侧加防御）

**问题**：门控触发完全依赖工作流 Skill 侧的自觉。

**优化**：在 `test-coverage-analyzer` SKILL.md 增加一段「合并门控场景提醒」：

> 当本 skill 被用于合并前门控场景时，若检测到当前分支已无未推送提交（即代码已合并），应输出警告「代码已合并，门控失去拦截价值，仅作留档参考」，提示调用方门控应在合并前运行。

这样即便工作流侧漏跑，test-coverage-analyzer 侧也能在补跑时给出明确提示。
